# RAG 与 Skill：AI 辅助配置类产品的核心方法论

> 源自 demo2（访问策略生成 skill）验证。**概念澄清 + 可迁移架构 + 落地指标**。

---

## 一、概念澄清：RAG vs Skill 的分工

这是两个常被混淆的概念，实际分工清晰：

| 组件 | 比喻 | 职责 | 不管什么 |
|------|------|------|---------|
| **RAG（检索增强生成）** | 外挂知识库 | LLM 生成回答前先检索外部库，解决"信息准确性"（URL、具体字段、业务规则） | 流程步骤 |
| **Skill（专业技能/工具包）** | 乐高说明书 | 让 AI 会做事、会调流程、会按最佳顺序执行动作 | 具体知识来源 |

**组合原则**：
- RAG 管"**信息准不准**"——防止 LLM 编 URL、编字段
- Skill 管"**流程顺不顺**"——防止每次都从零思考工具使用顺序
- 二者结合才是"即时可用的策略"

### 常见误区

❌ 用 prompt 喂一堆知识代替 RAG → 上下文爆炸 + 幻觉
❌ 让 LLM 自己决定每次工具调用顺序 → 同样需求不同结果
❌ 把对话历史当 Skill → Skill 是**静态固化制品**，不是对话记录

## 二、Skill 复用机制（重要认知纠偏）

这个细节在官方文档里不显眼，但直接影响产品设计：

- ✅ **一轮对话 = 生成 1 个技能**
- ✅ **多轮对话 = 优化同 1 个技能**
- ✅ **`skill.zip` 永远只有 1 套技能模板**
- ✅ **调用时只会命中这一个技能**

换句话说：无论用户和 AI 聊了几轮，最终固化下来的 Skill 制品都是一个。这决定了：

1. **Skill 设计要"可迭代"而非"可堆叠"**：新需求应优化旧 skill，不是平行建多个
2. **多轮对话是炼制过程，不是产物**：产物只有 JSON + skill 定义
3. **命名和版本管理要清晰**：因为只有一份，它代表"当前最佳实践"

## 三、AI 辅助配置类产品的五层架构

demo2 跑通后提炼的通用架构。**任何"自然语言 → 结构化配置"的产品都可套用**（访问策略/工单分派/审批流/报表筛选器…）。

```
用户输入（自然语言描述）
        ↓
┌──────────────────────────────┐
│ ① 对话引导层（LLM + RAG）     │ 理解意图、多轮追问、调用知识库
├──────────────────────────────┤
│ ② 信息收集与验证层            │ 提取关键字段、校验必填、处理模糊
├──────────────────────────────┤
│ ③ JSON 生成层                │ 应用默认规则、结构化输出、格式校验
├──────────────────────────────┤
│ ④ Skill 沉淀层               │ 保存对话模式、提取规则、更新 KB
├──────────────────────────────┤
│ ⑤ 系统集成层                 │ 同步策略系统、部署插件、记录日志
└──────────────────────────────┘
        ↓
生产就绪配置
```

## 四、两条处理路径：炼制 vs 复用

### 路径 A：第一次炼制（冷启动）

1. **RAG 知识检索**（自动）——关键词提取 + 向量匹配
2. **多轮对话参数收集**——典型 6 轮（以访问策略为例）：
   - 访问控制类型（禁止/审批/只读/限时）
   - 审批人确定（BOSS/上级/部门/多级）
   - 审批模式（实时/定期/预授权）
   - 临时权限时长（固定/审批人定/申请人填）
   - 申请理由要求（必填/可选/不要）
   - 生效时间范围（7×24/工作时间/非工作时间）
3. **JSON 生成** + 自动补齐企业级最佳实践（超时、多渠道通知、审计日志）
4. **Skill 沉淀**（触发关键词、决策规则、最佳实践、复用模板）
5. **RAG 库使用统计更新**

耗时 ~15 分钟，6 轮对话 + 1 轮确认。

### 路径 B：第二次复用（热路径）

1. **相似度匹配**（自动）——文本向量化 + 与 Skill 库比对
2. **快速复用确认**——展示匹配信息，用户选：
   - A 直接复用（推荐，置信度 ≥ 95%）
   - B 基于现有定制（80% ≤ 置信度 < 95%）
   - C 重新配置（置信度 < 80% 或用户主动选）
3. **配置验证 + 部署准备**

耗时 ~30-60 秒，1 轮确认。

### 关键阈值

| 置信度 | 系统行为 |
|--------|---------|
| ≥ 95% | 自动推荐"直接复用" |
| 80% - 95% | 推荐"基于现有定制" |
| < 80% | 走炼制路径 |

## 五、效率对比数据（验证复利）

同一需求在两次流程中的表现（来自 demo2 实测）：

| 指标 | 炼制 | 复用 | 改善 |
|------|------|------|------|
| 对话轮次 | 6 | 2 | ↓67% |
| 耗时 | ~15 min | ~2 min | ↓87% |
| 手动决策次数 | 5 | 1 | ↓80% |
| 自动化程度 | 20% | 90% | ↑350% |

**启示**：产品复利来自**把决策过程沉淀为决策模板**。用户第一次回答 5 个问题，第二次只需回答 1 个。

评估一条 AI 链路能否上线的统一口径：

| 指标 | 意义 | demo2 实测 |
|------|------|-----------|

## 七、未解问题与风险提示

- **知识库手动收集成本**：没匹配的页面靠用户添加+审核入库，这个闭环如果不通，RAG 退化为半自动百科
- **模糊/矛盾表述的兜底标准**：多轮对话触发时机、轮数上限需量化
- **部署复杂度**：大量 python 脚本对业务侧不友好，需封装成服务
- **测试环境选型**：本地跑通 ≠ 可部署，要对齐真实工单场景

## 八、ver2.3 新认知：LLM 自检索的"大类展开失败"与预设库兜底（2026-04-23 深化）

原本以为"RAG 数据完整 = URL 不会漏"。ver2.3 真实环境测试（13 条独立策略，85% 通过率）把这个假设打破了——**即使 RAG 219 条数据齐全，LLM 在"大类 → 子页面完整枚举"这类任务上仍会系统性遗漏**。

### 具体案例

需求："禁止进入设置下的所有页面"。

- "设置"大类下共 12 个子页面，RAG 库 219 条里全部有收录
- 实测结果：LLM 只关联到其中 8 个，漏了 `账户信息/礼品选项/税务设置/您的信息和政策` 4 个
- 漏掉的 4 个页面若被访问，整条"禁止设置"策略即被绕过

### 根因拆解

| 误以为的问题 | 实际问题 |
|------------|---------|
| RAG 数据不全 | RAG 没问题，219 条都在库里 |
| 检索算法差 | 不是单次检索问题，是 LLM 没意识到要做"穷尽式检索" |
| Prompt 没讲清 | 讲了也没用——LLM 做"大类展开"时容易"采样式回答"而非"枚举式回答" |

**本质**：LLM 的默认行为是"生成看起来合理的答案"，而非"穷尽所有相关答案"。面对"A 下所有 X"这种**集合闭包**类任务，LLM 的采样天性会偷懒。

### 修复思路：Skill 预设库兜底

不再依赖 LLM 自主检索，而是在 Skill 预设库中直接登记"高频大类 → 子项完整清单"的映射表。LLM 命中大类关键词时，直接取预设表，绕过自检索：

```
传统链路：
用户说"设置" → LLM 调 RAG 检索 → 返回相关 URL（可能漏）

预设兜底链路：
用户说"设置" → Skill 预设库命中"设置大类" → 返回 12 条完整 URL（不漏）
                                  ↓ 若未命中
                              回落到 RAG 检索
```

### 通用原则

| 情景 | 依赖 LLM 自检索 | 用预设库兜底 |
|------|----------------|-------------|
| 单点查询（"Amazon 的 case 页面在哪"） | ✅ 够用 | 不必要 |
| 大类展开（"设置下所有页面"） | ❌ 会漏 | ✅ 必须 |
| 同功能多入口（"假期设置"两条路径） | ❌ 易遗漏副入口 | ✅ 必须 |

**判别信号**：用户描述里含"所有 / 全部 / 整个 XX 模块 / XX 下的" → 触发预设库优先。

### 对五层架构的补丁

原"② 信息收集与验证层"补一条规则：**遇到大类关键词，先查 Skill 预设库的完整子项表，命中即绕过 LLM 自检索**。这是"Skill 沉淀层（④）"反哺"信息收集层（②）"的典型路径——Skill 不只沉淀流程，也沉淀"高风险失败点的打补丁数据"。

## 九、ver2.4 方向：知识库建设从纯人工到组合流水线（2026-04-24 深化）

原第七章"知识库手动收集成本"是 ver2.3 之前留下的未解问题。本章给出具体方案。

### 现状确认（本次新增事实）

直接咨询技术方确认了 ver2.3 知识库的真实构建方式：

| 组件 | 构建方式 |
|------|---------|
| **URL**（219 条） | 纯人工收集 |
| **name**（每条 URL 的页面名称） | 纯人工标注 |
| **别名**（平均 8.6 个/条） | **AI 基于 name 自动生成** |
| **关键词**（平均 11.9 个/条） | **AI 基于 name 自动生成** |

**瓶颈定位**：**URL 发现 + name 标注**是纯人工环节，语义扩展（别名/关键词）已经半自动。这意味着组合流水线的后半段其实跑通了，问题只在前半段。

隐含后果：
- URL 覆盖度受限于收集人的使用经验——**子页面系统性遗漏**（ver2.3 Bug A、第八章"设置大类漏 4 条"都是此因的表现）
- 人工方式**不可扩展**，无法支撑下一阶段"Amazon + TikTok + Walmart 三平台"的铺开

### 扩展性瓶颈的量化推理

| 维度 | 人工方式代价 |
|------|-------------|
| 初始建库 | 每平台 200-300 条 URL × 人工标注语义 = 1-2 人周 |
| 持续维护 | 平台改版后重做，每次 0.5-1 人周 |
| 人员依赖 | 必须懂业务的人做，难以众包 |
| 扩展到新平台 | 完全线性增长，一次性不可摊销 |

**结论**：人工方式在 ver2.3（单平台）是可接受的 MVP 代价，在 ver2.4+（多平台）是**产品化的首要工程瓶颈**。

### 方案评估对比

| 方案 | URL 源 | 语义源 | 跨平台成本 | 合规风险 | 综合评分 |
|------|:------:|:------:|:---------:|:-------:|:-------:|
| **爬虫** | 🔶 部分 | ❌ | ❌ 每平台重做 | 🔴 违反 ToS | ⭐⭐ |
| **紫鸟智能网页采集** | ✅ 真实用户路径 | ❌ 需补 | ✅ 一套通吃 | ✅ 自家功能 | ⭐⭐⭐⭐⭐ |
| **录屏 / 关键帧 AI 识别** | ✅ | ✅ AI 可描述页面用途 | ✅ 视频识别平台无关 | ✅ 无 | ⭐⭐⭐⭐ |
| **官方文档** | ✅ 完整但可能过时 | 🔶 部分 | ❌ 各平台自查 | ✅ 无 | ⭐⭐⭐ |
| **纯人工**（现状） | ✅ | ✅ | ❌ 线性增长 | ✅ 无 | ⭐⭐ |

### 爬虫方案被否决的理由（常被误判为"首选"）

- **Amazon Seller Central 反爬极严**：登录墙 + 设备指纹 + 动态 JS 渲染 + 状态依赖跳转，简单 HTTP 爬虫基本无效，headless browser 成本高且易触发封禁
- **违反 ToS**：爬 Amazon 可能导致测试账号被封，影响客户关系
- **跨平台不可复用**：TikTok Shop 反爬比 Amazon 更狠（字节系对抗能力强），Walmart 又是另一套，每个平台都要重做一套爬虫
- **不产生语义**：即便拿到 URL，也不知道它对应用户口中的"广告后台"/"推广数据"——还是要人工标注

### 推荐组合方案：URL 与语义分开采集

```
阶段 1：紫鸟智能网页采集 → 拿到真实 URL 清单
         ↓
阶段 2：录屏 / 关键帧 AI 识别 → 为 URL 补充页面语义描述
         ↓
阶段 3：AI 批量生成别名/关键词 → 变成可用于策略生成的知识库条目
         ↓
阶段 4：业务专家抽检 → 保证入库质量
```

**关键设计**：**URL 来源和语义来源解耦**。不指望一个方案 all-in-one。

- URL 问题归 URL 问题（紫鸟智能网页采集擅长）
- 语义问题归语义问题（AI 关键帧识别擅长）
- 人工只做最后的**抽检兜底**，不做初始生产

### 对五层架构的补丁

原架构"④ Skill 沉淀层"默认是人工维护。本方案让**④ 反向驱动"知识库数据供给管道"**：

```
原架构补丁：

[紫鸟智能网页采集] ──URL──┐
                          ▼
[录屏/关键帧 AI 识别]──语义─┐
                           ▼
                  ┌──────────────────┐
                  │ 知识库自动生成管道 │──→ 知识库（供 RAG 用）
                  └──────────────────┘
                           ▲
                  [人工抽检反馈回流]
```

这也回答了第七章"知识库手动收集成本"的最终方案——**从"纯人工"转向"自动化流水线 + 人工抽检"，而不是"纯人工"vs"纯自动"的二元选择**。

### 可复用的方法论原则

以下原则对任何"自然语言 → 结构化配置"类产品都适用：

1. **数据源和语义源可以解耦**——不要指望一个数据采集方案同时解决"有什么"和"它是干啥的"
2. **人工留在抽检环节**，不要留在初始生产环节——生产环节的人工不可摊销，抽检环节的人工边际成本低
3. **评估数据方案时要考虑跨平台成本**——在单平台看很划算的方案，扩展到多平台就暴露结构性问题
4. **合规风险要在方案对比时明确打分**——不是技术可行性的附注，是独立维度

## 十、核心结论

> 【自然语言描述 → 精确 URL 匹配（RAG） → 结构化策略输出（JSON） → Skill 复用加速】
>
> 这条链路已由 demo2 验证可行，且具备在其他"自然语言 → 结构化配置"场景迁移的能力。

**适用领域**（按迁移难度递增）：
1. 访问策略生成（本例）
2. 工单分派规则、审批流配置
3. 报表筛选器、监控告警规则
4. 业务流程自动化（需接入更多外部系统）
