# 产品思维方法论库

## 1. 三问法框架 ⭐⭐⭐⭐⭐

**核心概念：** 任何产品决策都应该先回答这三个问题

```
What?  ← 要做什么事情？（明确具体要打造的功能/产品是什么）
  ↓
Why?   ← 为什么要做这个事情？（挖掘核心价值、能为客户带来什么实际价值、解决什么问题）
  ↓
How?   ← 如何落实实现？（从表层交互体验到B端产品底层逻辑）
```

**实际案例：紫鸟浏览器的访问策略功能**

| 层级 | 问题 | 答案 |
|-----|------|------|
| What | 要做什么 | 构建灵活可配置的访问策略系统 |
| Why | 为什么 | 为BOSS提供精细化权限管理 + 约束员工非法操作 + 保障账号安全 |
| How | 如何实现 | 事前拦截 → 事中监管 → 事后追溯的三层安全体系 |

**应用时机：**
- PRD编写前的需求理解
- 功能设计的优先级评估
- 跨部门需求对齐的讨论

## 2. 产品心态转变 - 「从后妈变成亲妈」

**本质：** 从被动应付 → 主动负责的心态升级

### 三个关键转变

#### ① 自信度
- **后妈心态：** 被动接收需求，按要求完成即可
- **亲妈心态：** 对自己的产品有自信，敢于质疑不合理的地方

#### ② 展示方式
- **后妈心态：** 对外展示产品的优点和闪光点
- **亲妈心态：** 对内坦诚了解产品的缺点和短板

#### ③ 问题解决优先级
- **后妈心态：** 有问题？快速解决，冲出去
- **亲妈心态：** 首要目标是找出问题根源在哪，其次才是解决问题

**实践建议：**
- 定期审视产品的3大缺陷
- 每次用户反馈都要追问「为什么？」而不是「怎么改？」
- 对产品的疑问要记录下来，作为改进的线索

## 3. 多维度数据分析与对标

**应用场景：** 对标竞品、评估功能价值、优化成本

### 案例：录屏 vs 紫鸟浏览器的操作记录方案

**7个评估维度：**

| 维度 | 评估指标 | 仅录屏 | 紫鸟方案 | 产品启示 |
|-----|---------|-------|--------|--------|
| **存储成本** | 同等信息量下的空间占用 | ❌ 大 | ✅ 小 | 关键操作 > 完整录制 |
| **查询效率** | 从日志中找出特定操作的时间 | ❌ 极低 | ✅ 极高 | 可检索性 > 可视性 |
| **业务关联性** | 是否支持多维度筛选查询 | ❌ 无 | ✅ 有 | 数据结构化 → 分析工具化 |
| **数据价值** | 数据可用于其他业务分析吗？ | ❌ 有限 | ✅ 高 | 设计时考虑数据二次利用 |
| **数据安全性** | 防伪造和篡改的能力 | ❌ 低 | ✅ 高 | 多源数据互证 |
| **实现难度** | 技术实现的复杂度 | ✅ 低 | ❌ 高 | 权衡功能vs成本 |
| **用户体验** | 用户查看和理解的便利性 | ✅ 高 | 🟡 中 | 技术方案服务业务目标 |

**分析方法论：**
1. 列出关键的评估维度（与产品目标相关）
2. 定性或定量打分
3. 确定哪些维度是「必须优秀」vs「可接受」
4. 据此指导产品设计决策

## 4. 权责清晰化设计 - 多方诉求对齐

**适用场景：** 涉及多角色协作的B端产品（管理员-员工-平台）

### 核心设计理念

**目标：** 通过产品介入，将隐性的权责关系明确化，形成闭环

```
前：                          后：
     关系模糊                  权责清晰
┌─────────────┐           ┌──────────────┐
│   员工      │ ?         │   员工       │
│    ↔        │ ────────→ │    ↕ 规约   │
│   BOSS      │ (产品介入) │   BOSS       │
└─────────────┘           └──────────────┘
                                ↑
                            平台定义规则
```

### 三方诉求对齐框架

| 角色 | 核心诉求 | 痛点 | 产品设计 |
|-----|--------|------|--------|
| **员工** | 在权限范围内高效完成工作，避免误操作 | 不知道操作边界、容易犯错 | ①限定隔离环境②权限边界清晰③风险操作提醒 |
| **BOSS** | 实行管理权能，掌控账号安全，避免损失 | 无法实时掌握员工操作、事后追责困难 | ①自定义访问策略②风险操作实时审批③全链路日志审计 |
| **平台** | 构建B端安全闭环，适配平台合规要求 | 不同用户的需求差异大、难以标准化 | ①隔离环境+专属IP②三链路安全体系③权限分级一体化 |

**实施要点：**
- 事前拦截：提前限制不合理操作
- 事中监管：即时反馈和报警
- 事后追溯：完整日志支撑审计

## 5. 新人创新思维 - AI时代的创新点 ⭐⭐⭐⭐⭐

**难度等级：** ⭐⭐⭐⭐⭐

**挑战：** 在既有产品框架内体现个人创新能力

### 寻找创新点的方法论

**方法1：从「为什么不...」质疑开始**
```
观察 → 质疑 → 假设 → 验证 → 改进方案

例如：
观察：系统只有手动操作                  
质疑：为什么不加AI Agent自动回答用户问题？
假设：AI能回答80%的常见问题，提升自助率
验证：调研用户常见问题，估算解放成本
改进：提出"AI客服模块"的PRD
```

**方法2：从成本模型的不合理开始**
```
观察 → 质疑成本逻辑 → 提出替代方案 → 对标方案

例如：
观察：远程服务功能按时间计费（20min）
质疑：为什么不按次数/按数据量/按成本平摊？
对标：看其他SaaS如何定价
方案：提出更合理的成本模型
```

**方法3：从概念的内在逻辑开始**
```
现象 → 追问关系 → 梳理逻辑 → 优化架构

例如：
现象：设备/账号/安全三个模块并行
追问：它们真的是并行吗？有没有依赖关系？
逻辑：账号绑定设备 → 安全基于设备的隔离
优化：能否整合成"设备-账号-安全"的一体化架构？
```

**💡 核心洞察：**
创新 ≠ 凭空造新概念  
创新 = 在深入理解现状基础上的「不满足」+ 「想象力」

## 6. 业务流程图 vs 功能流程图

**容易混淆的地方：** 两者都是流程图，但服务的目标不同

### 对比表

| 维度 | 业务流程图 | 功能流程图 |
|-----|----------|----------|
| **视角** | 用户角度 | 系统角度 |
| **问题** | 用户要做什么？ | 系统如何支撑？ |
| **内容** | 用户的操作步骤 | 系统功能之间的逻辑依赖 |
| **例子** | 开始→租用服务器→日常运营→安全监管 | 工作台→账号环境→浏览器→日志记录 |
| **作用** | 指导产品设计优先级 | 指导开发实现逻辑 |

**实际应用：**
- **先画业务流程图** → 理解全局用户旅程
- **再画功能流程图** → 理解系统如何支撑这个旅程
- **对齐两者** → 确保功能设计服务于业务目标

## 7. 产品功能优先级的迭代思路

**关键认知：** 初期功能不必满足大多数用户，满足一部分即可，层层递进扩大范围

### 优先级决策框架

```
需求池
  ↓
【试点评估】
├─ 试点有效 
│   ├─ 数据支撑（核心指标达成）
│   ├─ 用户反馈（正反馈 > 75%）
│   └─ 继续推进 → MVP → 优化迭代 → 规模化
│
└─ 试点无效
    ├─ 原因分析（产品设计缺陷？市场不需要？）
    ├─ 决策路口：保留or改进or移除
    └─ 战略层审视（是否与长期方向对齐）
```

### 权衡优先级的三大原则

1. **与产品战略方向相关** - 确保功能往正确的方向发展
2. **有定量数据支撑** - 而非团队主观拍脑袋
3. **团队认知对齐** - 避免推进过程中频繁的方向调整

**警惕的陷阱：**
- ❌ 功能太多，用户不知道用哪个
- ❌ 试点失败了还坚持做
- ❌ 只做加法，不做减法

## 8. 假设验证：带着假设去问，不带着问题去问 ⭐⭐⭐⭐

**核心概念：** PM 在跨职能协作中获取信息时，提问方式直接决定信息密度

### 两种提问方式对比

| 提问方式 | 示例 | 得到什么 |
|---------|------|---------|
| **开放问题**（带问题去问） | "URL 是怎么收集的？" | 对方随便答，可能简化措辞、跳过细节，信息含糊 |
| **假设验证**（带假设去问） | "URL 是不是人工收集的？" | 对方要么确认、要么纠正，**信息密度和决策价值高得多** |

### 实际案例：从"默认"里发现瓶颈

**背景**：demo2 知识库有 219 条 Amazon URL，项目介绍里从没说是怎么来的——"默认"是自动的。

**PM 动作**：带着假设"是不是人工收集的"去问技术同学。

**得到信息**：确认是**纯人工收集**。

**解锁的洞察**：
- 子页面缺口不是偶然，是人工方式的**结构性短板**
- 扩展到 TK/Walmart 三平台如果继续人工，工作量不可持续
- 这是产品化阶段的**首要工程瓶颈**，必须纳入架构设计

如果没问这一下，这个瓶颈会拖到真正扩展时才暴露——那时工期已经紧了。

### 日常可以多问的 4 类问题

对项目里**每一个"自动"或"默认"的事**，带着好奇去问：

1. **这个数据是怎么来的？**（人工 or 系统 or 第三方 or AI）
2. **这一步是人做还是系统做？**（人做的地方就是未来的自动化空间）
3. **如果量级大十倍还能维持吗？**（暴露扩展性瓶颈）
4. **万一这个人离职了，能接手吗？**（暴露知识单点依赖）

看似在问技术实现，**实际上是在识别产品未来的脆弱点**。很多产品的隐性债务就是在"默认"和"当然"里积累的。

### 和"后妈心态 → 亲妈心态"的关系

第 2 条方法讲"亲妈心态"——主动了解产品的缺点和短板。假设验证就是**把亲妈心态落地到日常工作**的一个具体动作：

- 后妈心态 → 听见什么是什么，不质疑
- 亲妈心态 → 对"默认"和"想当然"保持敏感，主动验证

### 适用时机

- 接手新项目时（快速摸清底牌）
- 评估扩展可行性时（发现隐藏瓶颈）
- 跨职能需求对齐时（挤出含糊信息）
- 设计产品化方案时（区分"已验证"vs"未验证"）

### 警惕的误区

- ❌ 假设验证 ≠ 审问/质疑人。问法要让对方感到"你在思考"，不是"你在怀疑他"
- ❌ 不要带着错误假设反复暗示对方，得到顺口的"是"就完事——要听对方是否主动补充细节
- ❌ 不适合所有场景：完全陌生的领域还是要先开放问，建立基本认知之后再切换为假设验证

## 9. 生成能力 vs 理解能力 —— AI 能力评估的二元分层 ⭐⭐⭐⭐⭐

**核心概念：** 评估 AI 产品时，"能生成"和"真理解"是两回事；验收标准要拆成两层，不能混着测。

### 二元分层

| 能力层 | 它做什么 | 验收方式 | 决定什么 |
|------|---------|---------|---------|
| **生成能力** | 知道"这里该用通配符，那条应该限制访问" | 看输出格式/字段是否可用 | demo 能不能**用** |
| **理解能力** | 知道"这个岗位为什么屏蔽广告页，那个场景为什么只保留订单页" | 反推意图、盲测配置的"为什么" | demo 能不能**走远** |

**关键洞察：** 生成能力的天花板是"看起来对"，理解能力的天花板才是"替代人类专家"。

很多 AI 产品只测了生成能力，就以为自己理解了业务——其实只是记住了正确答案的形状。

### 验证生成能力 ≠ 验证理解能力

**生成能力测试**："用户说 X → AI 输出什么 JSON？" —— 正向输入 → 正向输出
**理解能力测试**："给 AI 看一份 URL 清单，让它反推这个策略是给谁用的、为了什么" —— 反向推断

反推比正推更能暴露 AI 是不是"真懂"。如果 AI 能看着 40 条真实策略的 URL 清单，准确反推出"这是给运营用的黑名单策略"，才叫理解。

### WHAT / HOW 双要素盲测打分法

**场景：** 40 条真实企业访问策略，隐藏原策略名，让 AI 只看 URL 清单重新起名 + 选适用成员。

**起名相似率的拆解**（每条满分 2 分）：

| 要素 | 含义 | 举例（原名"仅访问加拿大"） |
|------|------|-------------------------|
| **WHAT** | 这条策略管什么（业务对象/页面领域/功能模块） | 加拿大站 |
| **HOW** | 这条策略怎么管（控制方式/动作） | 仅访问（白名单逻辑） |

打分示例：
- AI 输出"加拿大站点白名单策略" → WHAT ✓ + HOW ✓ → **2/2**
- AI 输出"加拿大访问限制" → WHAT ✓，但"限制"与"仅访问"语义相反 → **1/2**
- AI 输出"站点管理策略" → 都没命中 → **0/2**

公式：**起名相似率 = 40 条总得分 ÷ 80 × 100%**

**成员命中率**（二元判定）：12 选 1（运营/财务/仓储/申诉/美工/客服/老板/临时外包/新员工/测评岗/全员除老板/都不匹配），选对 = 1，选错 = 0。公式：**成员命中率 = 命中条数 ÷ 9 × 100%**（只对原名明确含成员的 9 条计分）。

### 广度 × 深度 的双轮验证骨架

| 轮次 | 测什么 | 方法 | 规模 |
|------|-------|------|-----|
| **广度** | AI 看正常数据能不能理解 | 隐藏原策略名，让 AI 反推 | 40 条真实策略 |
| **深度** | AI 面对残缺数据能不能指出漏洞 | 人为删掉关键 URL，看 AI 能否察觉并说明 | 12 条漏洞注入 |

只做广度测试 → 测的是"AI 能不能跟着数据走"；加上深度测试 → 测的是"AI 有没有建立业务直觉，敢反驳数据"。

### 可迁移场景

这套 **反推 + 双要素 + 广深双轮** 的评估骨架，适用于任何"AI 生成结构化配置"类产品：

- 工单分派规则（反推这条规则想分派给哪类工单）
- 审批流配置（反推这条流想卡住什么风险）
- 报表筛选器（反推这组筛选器想看什么业务问题）
- 客服意图识别（反推这类话术对应什么用户意图）

### 与既有方法论的衔接

- 第 2 条**亲妈心态** → 不满足于"看起来能用"，要主动测"是不是真懂"
- 第 8 条**假设验证** → 反推验证本身就是带假设去问 AI"你是不是理解了这个"

## 10. AI 产品的能力边界自知 —— 不输出对接系统不支持的方案 ⭐⭐⭐⭐

**核心概念：** AI 生成的配置方案不能超越对接系统的实际能力，否则用户收到"看起来完美但无法落地"的答案，比直接报错更坑。

### 问题场景（真实案例）

用户需求："设置访问策略，给发货专员限制除了库存相关的所有页面"

AI 输出的问题：
1. 生成的策略里用了"**可访问**"模式，但紫鸟平台**实际没有这种模式**——用户被误导
2. 在只能使用黑名单的情况下，大量页面没屏蔽（促销、定价、品牌分析、退货、店铺、账户健康等）——覆盖不全

**问题总结：能力边界定义不清晰**。AI 不知道自己对接的平台"能做什么、不能做什么"。

### 解法：平台能力地图 + 指路规则

在 Skill 里显式写明：

```
┌─────────────────────────────┐
│ 已覆盖能力（AI 可输出）      │  URL 级页面屏蔽
│                             │  URL 级可访问+行为控制
├─────────────────────────────┤
│ 未覆盖能力（AI 应拒答并指路） │  白名单、访问特例、元素级屏蔽
└─────────────────────────────┘
```

**指路规则**：用户需求落在"未覆盖能力"区时，AI 不硬凑方案，而是**明确告知"此需求超出当前能力边界，建议走平台原生功能 XX"**。

### 为什么这件事容易被忽略

AI 模型本身知识广，写代码/写配置时总能"编一个合理答案"。但在 **对接特定系统** 的场景下，"合理"不等于"可用"——平台支持哪些模式是硬约束。

让 AI 自知边界的成本不高（写一张能力地图），不做的代价却很高（用户投诉 + 信任崩塌）。

### 设计原则

| 场景 | 对应设计 |
|------|---------|
| AI 的输出要进对接系统的表单 | 字段名、枚举值、字段间联动规则全部显式声明 |
| AI 要说"我不会" | 事先定义哪些问题超出能力，给出"指路话术" |
| 系统能力会升级 | 能力地图版本化管理，升级时同步更新 Skill |

### 可迁移场景

任何"AI 输出 → 系统执行"类产品：低代码平台的可视化生成、BI 的仪表盘生成、客服工单的分派建议、代码编辑器的补全——**AI 的输出空间必须被目标系统的能力空间所包含，不能越界**。

**源头：** SKILL 进度及规划汇报文档（2026-04-30），demo2 v2.2 的"能力边界 + 平台地图"迭代与反馈表 1 号 bug。

## 11. LLM 硬约束的四级强度梯度 ⭐⭐⭐⭐⭐ 新增

**核心概念：** LLM 对不同强度约束的响应率差一个量级。设计 Skill 提示词时按梯度从弱到强上行，**只在上一级失效时才升级**，不要一开始就用最强约束。

### 四级梯度表

| 级别 | 话术形式 | AI 执行率（经验值）| 适用 |
|---|---|---|---|
| 1 · 软提示 | "应该先..."、"建议..."、"可以考虑..." | ~30% | 友好建议，不强制 |
| 2 · 原则约束 | "必须先..."、"不得..."、"禁止..." | ~60% | 基础规则 |
| 3 · 固定话术 | "第一句必须按此句式开头：我匹配到《X》..." | ~90% | 关键输出 |
| 4 · 禁止清单 + 合法清单对照 | ❌ 禁止词：XXX / ✅ 合法词：YYY | 接近 100% | 红线级 |

### 实战案例：demo2 v3.0 Phase 1.5 加固（2026-05-07）

Phase 1.5 模板匹配能力从"时灵时不灵"到"22/22 命中"的过程，本质上是约束强度梯度升级的过程：

- **第 1 轮（级别 2）**："必须匹配模板并宣告结果" → AI 偷懒不 Read 模板文件，或者 Read 了不宣告
- **第 2 轮（级别 3）**："第一句必须按此句式开头：我匹配到《X》..." → 命中时稳定开场，但短模糊需求下 AI 仍滑到 RBAC
- **第 3 轮（级别 4）**："❌ 禁止 RBAC/ERP/HR/CRM 词 / ✅ 合法紫鸟模块词对照" → 红线守住，红线触发 0 次

每升一级都是对上一级失败 case 的回应。**不要一上来就用级别 4**，会牺牲 AI 对合理变通的灵活性；也不要停在级别 2 不动，对关键输出力度不够。

### 深化已有框架

这条方法论是第四周"**Skill 提示词工程制品**"概念的操作化——把"Skill 是指令集"这件事拆到执行层：指令有强度梯度，按需使用。

**源头：** 本次对话（2026-05-07）demo2 v3.0 Phase 1.5 模板匹配加固过程的 4 轮实战。

## 12. 测试的三层证据结构（覆盖率 × 多样性 × 一致性） ⭐⭐⭐⭐⭐ 新增

**核心概念：** 验证 AI 能力不能只看"跑对几条"。要分三层独立论证——**只有三层同时成立**，才能说"AI 学会了"。

### 三层结构

| 层 | 测什么 | 数据产出 | 样本量 | 时间成本 |
|---|---|---|---|---|
| Layer 1 · 覆盖率 | 算法覆盖的范围边界 | "测过 N% 的能力项" | 全量 | 可脚本化 |
| Layer 2 · 多样性 | 每个类别都 work | "所有类别抽样命中率 X%" | 每类 1-3 条 | 小时级 |
| Layer 3 · 一致性 | 同题多次跑稳定 | "关键题连跑 3 次一致率 Y%" | 关键题 × 3 次 | 小时级 |

### 组合解释力

- **只有 Layer 1 没 Layer 3**：覆盖率高但每次跑结果不同 → 不能产品化
- **只有 Layer 3 没 Layer 1**：稳定但盲区大 → 尾部风险未知
- **Layer 2 是最佳性价比**：18-22 条覆盖多样性，小时级完成，能支持"主干能力验证"陈述
- **三层同时成立**才能说"AI 学会了"

### 实战案例

demo2 v3.0 Phase 1.5 模板匹配验收（2026-05-07）：
- Layer 1 覆盖率 38.6%（22/57 已测 + 35 条未测）→ 不支持"完全学会"
- Layer 2 多样性 22/22 命中（100%）→ 支持"主干能力验证通过"
- Layer 3 一致性 93.4%（5 关键题 × 3 次）→ 支持"稳定运行"

**对外陈述选择**：用"MVP 主干能力验证通过"，不用"AI 学会了"——因为 Layer 1 数据不够。这避免了被追问"那剩余的 35 条呢"时无话可答。

### 深化已有框架

第五周"**意图验证方法论选择矩阵**"（方法论 5）选的是"测什么方法"，这条方法论选的是"**用多少层证据证明**"。两个方法论配合使用：
- 先用方法论 5 决定测法（盲测 / 漏洞注入 / 留一 / 扰动）
- 再用方法论 12 决定分多少层做

### 适用场景

反推实验、RAG 评估、Skill 行为验证、任何"AI 能不能做 X 类任务"的判断。

**源头：** 本次对话（2026-05-07）demo2 v3.0 模板匹配测试分层设计。

## 13. LLM 语境漂移的识别与防范 ⭐⭐⭐⭐ 新增

**核心概念：** LLM 面对**超短 / 模糊**需求时，会滑向**训练分布里最常见的模式**——即便你的 Skill 已经限定了领域。这种"语境漂移"是 AI 产品设计最隐蔽的坑。

### 识别信号：输出里出现当前产品域不存在的词汇

**demo2 实测案例**：

| 用户输入 | 预期（紫鸟访问策略）| AI 实际输出（脱轨）|
|---|---|---|
| "财务的权限" | 紫鸟模板匹配 + 紫鸟模块追问 | ERP 总账 / 银企直连 / HR 薪酬 / CRM / 代码仓库 |
| "运营做个权限" | 紫鸟店铺运营模板匹配 | 用户数据 / 订单交易数据 / 内容管理 / 业务数据 |

共同点：输出里都是**跨行业企业权限管理概念**，和紫鸟浏览器的 Amazon/TikTok 页面访问控制**完全无关**。

### 底层原理

LLM 的"训练分布引力"：
- 用户说"财务权限" → 训练语料里"财务权限"最常见搭配是企业 ERP/HR/财务系统
- 用户没提"紫鸟"/"Amazon"等锚点词 → AI 默认回到这个分布最浓的区域

正向引导（"你是紫鸟助手"、"只处理访问策略"）力度有限——LLM 能理解这些原则，但不会主动对齐到最严格的解读。

### 防范：负向约束 + 正向锚定双清单

在 Skill/system prompt 加一张"**输出锚定**"章节，显式写两份清单：

```
❌ 禁止出现的跨域词：
   RBAC / ERP / 总账 / 应收应付 / 银企直连 / HR 薪酬 / 
   员工花名册 / CRM / 代码仓库 / 设计稿 / 研发资产...

✅ 合法的本域词：
   生效成员 / 生效平台账号 / URL 规则 / mode / behavior_controls / 
   Amazon Seller Central 金融/账户设置/订单/广告 / 
   TikTok Shop 订单/财务/商品/Compass/Affiliate...
```

**两份清单放在一起做对照**比单独列其中一份效果强——AI 能通过对比理解"哪些是本域词"。

### 可迁移场景

任何**领域窄、需要强边界感**的 AI 产品：
- 垂直客服机器人（金融/医疗/法律）
- 企业内部知识库 AI
- 特定系统的配置助手（本项目类型）
- 垂直分类器（避免回到通用多类分布）

### 深化已有框架

方法论 10"AI 产品的能力边界自知"（2026-04-30 沉淀）关注的是 **能力边界**（AI 能不能做 X），本方法论关注的是 **语境边界**（AI 输出的词汇体系是不是本域）。

两者关系：
- **能力边界**守住"AI 该不该做"
- **语境边界**守住"AI 做的时候用不用本域词"
- 两者结合才能保证输出不脱轨

**源头：** 本次对话（2026-05-07）demo2 v3.0 Phase 1.5 测试中"财务/运营的权限"两次脱轨事件诊断与修复。

## 14. MVP 路径的 YAGNI 判断——信任原数据的语义信号 ⭐⭐⭐⭐ 新增

**核心概念：** 做 AI 理解层（匹配、推荐、分类）时，最大的反直觉判断是——**不要一开始就给原数据全量补前置标注**。先跑一版用原数据看看，多数情况下语义信号已经够用，补标注是 YAGNI 典型。

### 判断框架

面对"要不要给数据补 what / how / 关键词 / 别名 / 标签"时，按以下顺序问：

1. **原数据里现有字段已经提供了多少语义信号？**
   - 人工填写的 name/title 字段 → 通常含核心概念
   - 人工填写的 description/doc 字段 → 通常含应用场景
   - 关联数据的可反查结构 → 可以做二次语义召回
2. **三者加起来够用就不补**
3. **不够用再补最少的那条**（比如只给命中率差的 5 条补，不是全量）
4. **绝不一开始就"全量前置标注"**

### 实战案例：demo2 模板匹配（2026-05-07）

开工前决策：要不要给 62 个模板补 `what / how / 匹配关键词 / 匹配别名`？

|  | 全量补标注 | 不补（MVP 路径） |
|---|---|---|
| 工时 | 60 × 10 分钟 = 10 小时 | 0 |
| 准确率乐观提升 | +5-10 个百分点 | 基线 |
| 冗余度 | 高（name/desc 已含大部分信息）| 低 |
| 迭代成本 | 模板增加要重新标注 | 0 |
| **实测结果** | 未测 | **Layer 2 22/22 命中** |

判断：**原生信号已经够强**——模板名（"财务人员访问限制"）包含核心角色/功能词，描述（"限制财务人员不能访问亚马逊其他无关页面"）包含应用场景，URL 反查 RAG 映射到中文描述和别名。三者加起来的子串打分已经稳定达到高置信阈值。

### 深化已有框架

方法论 5.5（第五周）"**方法论设计前要先看数据分布**"（WHO 79% 占位符教训）是同一家族的方法论：

- 第五周：**看数据再设计维度**（避免设计出 79% 无信息量的维度）
- 本周：**看数据再补标注**（避免 10 小时冗余标注）

共同点：**你担心的事可能本来就不存在**，先跑一版再看。

### 可迁移场景

- RAG 知识库建设（别先给每条补全量 metadata，先跑简化版看召回率）
- 分类器训练（别先给每条补丰富标签，看原 label 是否足够）
- 推荐算法（别一开始就用复杂特征工程，看 baseline 单字段匹配能到什么水平）
- 任何"AI 理解层"的数据准备

### 反面教训

如果原数据真的没语义信号（比如 name 是 `T-001/T-002` 这种无意义 ID），YAGNI 不适用——那时候补标注是必要的。**判断标准是"原数据是否人工填写 + 是否包含业务语义"，不是主观乐观还是保守。**

**源头：** 本次对话（2026-05-07）demo2 v3.0 CSV 机械导入 vs 全量标注的决策复盘。

## 15. 静默失败 API 的诊断三原则（假设驱动 / 单变量对照 / 独立校验） ⭐⭐⭐⭐⭐ 新增

### 定义
提交 API 返回 `ret:0, success`，但实际效果不符合预期（字段未入库、配置被静默裁剪、默认值覆盖）。这类"非报错型失败"是后端 API 对接里最贵的 bug，因为它**把"提交成功"和"意图达成"脱钩了**。

### 三原则

**原则 1 · 假设驱动**
每个诊断步骤必须明确提出**可证伪的假设**："如果 X 是真因，则实验 Y 应该让现象变化；如果 Y 没变化，X 就被证伪"。
反例：只解释现象（"这可能是因为...也可能是因为..."）→ 没有停止条件，无限延伸。

**原则 2 · 单变量对照**
复制上一轮失败的 payload，只改**一个字段**，其他完全照抄。连续测试 3-5 轮后差异点被机械定位。
反例：一次改 3 个字段"看看哪个有效"→ 回归不可解释。

**原则 3 · 独立校验**
提交后立刻用**独立读接口**（detail / list）回查实际持久化的值，不信 add 返回的 `ret:0`。
反例：只看 add 返回就宣告成功 → 几小时后才发现 configs 是空的。

### 本周实战（2026-05-09 访问策略 API 对接）

| 轮次 | 假设 | 单变量实验 | 结果 |
|---|---|---|---|
| ① | `configs[].id` 是非法值 | 删掉 id 重提 | 仍空 → 证伪 |
| ② | 黑白名单模式级冲突 | 扫全量规则对照 | 156 黑名单共存 → 证伪 |
| ③ | `preset_type` 可通过输入控制 | 传 `preset_type:1` | 被后端覆盖回 0 → 证伪 |
| ④ | url_id 无效 | 换百度 url_id | configs 完整入库 → 成立 |

四轮耗时 2 小时，比"一次改三个字段瞎蒙"效率高一个数量级。

### 为什么这比"想清楚再测"更有效

诊断静默失败时，脑子里能想出的"合理解释"几乎总是多余的。**解释成本 ≤ 实验成本**时（API 调用是毫秒级），直接进入"设计单变量实验"环节比"先穷举所有可能"收敛得更快。解释只在实验之前**提出可证伪假设**那一刻有用——之后实验说话。

### 适用场景
- 任何后端接口对接 / 第三方系统联调
- 任何"提交成功但行为异常"的诊断
- 任何"在别人机器上能跑，我这里不行"类型的问题

### 与方法论 8（假设验证）的关系
深化已有框架。方法论 8 讲"带着假设去问**用户**"，本方法论讲"带着假设去问**系统**"——前者用对话验证，后者用实验证伪，两者面向不同对象但共享"假设→验证→推翻/接受"的核心骨架。

**源头：** 本次对话（2026-05-09）访问策略 `/access_rule/add` 接口 configs 静默丢弃的 4 轮诊断。

## 16. 跨环境 ID 污染的识别与防范 ⭐⭐⭐⭐ 新增

### 定义
从环境 A（例：生产抓包、历史文档、其他租户）拷贝一个 ID，直接在环境 B（例：dev）里使用，而 B 根本没有这个资源。后端通常不做跨环境校验 → 行为看似合法，实际静默失败。

### 典型特征（符合 3/4 以上即高度怀疑）
- ID 是从**别处**（生产抓包 / 文档样例 / 同事发的聊天记录）复制来的
- ID 格式**无拼写错**（如 17 位时间戳、UUID、自增数字），看起来完全合法
- 后端接受该 ID 不报错，但相关字段**未生效**
- 同环境其他 ID 工作正常 → 排除了 API 整体故障

### 识别 checklist

```
① 这个 ID 是在当前环境注册的吗？
② 能在当前环境的 /list 或 /detail 接口里查到吗？
③ 如果答不上来 → 先调 list/detail 验证，再使用
```

### 本周实战

`url_id=17237191297447`（mooncell.com）——从**生产环境** hqyTest 抓包来的，dev 环境里没这个资源。所有用它建的规则 configs 都被后端静默丢弃。诊断走了 4 轮假设才绕回这个根因——**根源其实在"复制 ID 的那一刻"就埋下了**。

### 预防习惯
- 文档 / memory 里记 ID 时，**必须标注环境来源**（dev / prod / 哪个租户）
- 从别处复制 ID 使用时，**先在目标环境 list/detail 验证存在**
- 跨环境 pipeline（如 dev → prod 迁移）的 ID 映射必须做**显式翻译表**

### 适用场景
- 任何多环境系统（dev/staging/prod）
- 多租户 SaaS 系统
- 微服务跨服务 ID 传递
- 数据迁移 / 冷启动 bootstrap

### 元认知层扩展
这条可以抽象为"**边界透明性**"原则——任何跨边界传递的标识符都要携带"我来自哪"的元信息。当边界不透明时（一串看起来合法的数字），debug 成本会被放大到跨越边界才发现。**把边界显式化**（给 ID 加环境前缀 / 加 Namespace）是比"事后诊断"更优的预防手段。

**源头：** 本次对话（2026-05-09）mooncell url_id 从生产抓包流入 dev 环境，导致 4 轮诊断绕远。

## 17. 从前端抓包逆推后端模块路径的通用规律 ⭐⭐⭐⭐⭐ 新增

### 适用条件
- 没有后端 API 文档
- 不能直接问开发（时间紧 / 开发资源被卡）
- 需要对接某个功能（做集成 / 写自动化脚本 / 反向工程）

### 三段式操作

**① 前端抓包取 URL 样本**
- F12 → Network → 筛 Fetch/XHR
- 触发目标功能（点按钮、提交表单）
- 记下请求 URL：一般形如 `/api/v5/security/<module>/<action>`

**② 去版本前缀，裸 POST 后端**
前端路径经过代理/网关层，带有版本前缀（v1 / v5 等）。dev 直连后端时，去掉版本段：
```
前端抓包:   /api/v5/security/access_rule_url/add
dev 裸调:  /api/security/access_rule_url/add
```

POST 一个空 body：
- 返回 `{"raw":"no module"}` → 路径错，换模块名或前缀
- 返回 pydantic validation 错误（`field required`）→ **路径对了**，按错误提示补字段

**③ 按模块名穷举动作**
常见动作：`list / add / detail / update / delete`。挨个空 body 打，看 pydantic 错误清单反推每个动作的必填参数。

### 本周实战（2026-05-09）

| 步骤 | 动作 | 结果 |
|---|---|---|
| 1 | 拿到前端抓包 `/api/v5/security/access_rule_url/add` | —— |
| 2 | 裸调 `/api/v5/security/access_rule_url/add` | no module（错的） |
| 3 | 去掉 v5 裸调 `/api/security/access_rule_url/add` | pydantic 报 `name/desc/url_pattern` 必填 ✓ |
| 4 | 穷举：`/list`（要 page/limit）、`/detail`（要 id）、`/to_regex`（要 urls）、`/delete`（要 url_ids） | 4 个接口整个模块解锁 |
| 5 | 闭环验证：API 注册 URL → 拿 url_id → 建规则 → detail 查 configs 入库 | ✓ |

从"盲试 30+ 候选路径全部 no module"到"获得抓包后 5 分钟解锁整模块"的效率飞跃。

### 为什么"去版本前缀"是关键

前端/代理层加版本前缀的常见原因：
- 多版本 API 共存（v1 兼容老客户端，v5 支撑新客户端）
- gateway 用版本段路由到不同后端服务
- 代理层做统一鉴权/限流

**dev 环境往往绕过了代理层直连后端**（方便开发调试），所以路径要去掉代理层的"语义前缀"。这是一条经验规律，不是 spec 保证——不适用时需要换其他前缀试。

### 局限性
- 加密信封（生产环境常见）会挡住这条路——只能在 dev/明文环境有效
- 如果后端对 HTTP method、Header、Token 有额外要求，裸 POST 可能直接 401/403
- 模块名不规范（如 camelCase / kebab-case / snake_case 混用）的系统，穷举成本更高

### 适用场景
- 老系统没 API 文档要对接
- 业务急需某个接口但开发资源卡住
- 反向工程 / 迁移 / 自动化脚本 bootstrap
- 快速评估某功能的 API 成本

### 与方法论 8（假设验证）的关系
同一家族，都是"用最小代价换最快反馈"：发一个空 body 请求，pydantic 错 vs no module 是两种极清晰的信号，比读源码 / 逆向代码快一个数量级。
