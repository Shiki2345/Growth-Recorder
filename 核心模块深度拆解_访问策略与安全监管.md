# 核心模块深度拆解

## 模块1：访问策略 (Access Policy)

### 1.1 功能定位
**本质**：将企业的管理规则从人脑（BOSS的口头要求）显式化为系统可执行的数据配置。

**价值**：
- BOSS 无需改代码，即可实时调整安全策略
- 规则变更可灰度、可回滚、可追溯
- 支持多策略组合（白+黑+特例）

---

### 1.2 业务场景拆解

#### 场景1：白名单型策略
**描述**：某类操作只有特定员工可做

**配置示例**：
```
策略名称：亚马逊退款-仅BOSS可操作
应用范围：所有亚马逊账号
规则：
  - 操作类型 = 退款处理
  - 行为 = 白名单
  - 授权对象 = [BOSS]
  - 其他人触发：立即拦截 + 系统预警
生效时间：永久
优先级：1（最高）
```

**系统行为**：
```
员工操作流程 → 规则引擎检查 → 命中白名单规则 → 直接拦截
BOSS操作流程 → 规则引擎检查 → 白名单通过 → 继续执行
```

#### 场景2：黑名单型策略
**描述**：某类操作所有人都禁止（或需特殊审批）

**配置示例**：
```
策略名称：商品删除-全员禁止（需BOSS审批）
应用范围：所有账号
规则：
  - 操作类型 = 删除商品
  - 行为 = 黑名单（触发审批流）
  - 审批流程 = [发起] → [BOSS审批] → [执行 OR 拒绝]
  - 审批超时 = 24小时（自动拒绝）
生效时间：永久
优先级：2
```

**系统行为**：
```
员工触发删除操作 → 命中黑名单 → 操作冻结 → 发送BOSS审批通知 → 
等待BOSS在1小时内审批 → 审批通过：继续执行 / 审批拒绝：取消操作
```

#### 场景3：访问特例（Exception）策略
**描述**：临时豁免某个黑名单规则，仅对特定人在特定时间有效

**配置示例**：
```
策略名称：紧急授权-改收款方式
主策略：改收款方式-全员禁止
特例配置：
  - 员工：销售主管 李四
  - 账号：TikTok Shop 店铺A
  - 操作：改收款方式
  - 有效期：2026-04-20 09:00 ~ 2026-04-20 17:00（8小时有效）
  - 原因：紧急处理客户退款，需临时变更收款主体
  - 授权者：BOSS（CEO）
```

**系统行为**：
```
李四在有效期内访问TikTok账号 → 触发"改收款"操作 → 
检查特例规则 → 命中有效期内的特例 → 白名单通过 → 继续执行
（同时记录：操作者、操作时间、操作内容、授权者、授权理由）

有效期过期后 → 特例失效 → 恢复黑名单状态
```

### 1.3 多策略冲突处理

**冲突场景**：同一操作同时命中多条规则

**优先级决策树**：
```
规则匹配结果
├─ 白名单命中？
│  ├─ YES → 直接放行（优先级最高）
│  └─ NO → 继续检查
├─ 访问特例命中？
│  ├─ YES → 直接放行（优先级次高，且需验证有效期）
│  └─ NO → 继续检查
└─ 黑名单命中？
   ├─ YES → 触发审批流 / 拦截（优先级最低）
   └─ NO → 允许操作（无规则命中 = 允许）
```

**案例演演**：
```
场景：销售主管李四在2026-04-20 15:00 改收款方式

规则库：
Rule 1（白名单）: 金融主管+改收款 → 直接通过
Rule 2（特例）: 李四+账号A+改收款+2026-04-20 09:00-17:00 → 白名单通过
Rule 3（黑名单）: 全员+改收款 → 需BOSS审批

李四的身份：销售主管（非金融主管） → Rule 1 不匹配
时间验证：15:00 在 09:00-17:00范围内 → Rule 2 匹配 ✓
操作触发 → 规则引擎：白名单规则通过 → 直接允许执行
结果：操作成功，记录"特例规则授权"标签
```

#### 规则执行引擎伪代码
```python
def evaluate_access_policy(operator, account, operation):
    """
    评估访问策略，返回: (allow/reject/need_approval, reason, approval_config)
    """
    
    # Step 1: 获取该操作相关的所有规则
    relevant_rules = get_rules_for_operation(operation)
    
    # Step 2: 按优先级顺序评估（白 > 特例 > 黑）
    for rule in sort_by_priority(relevant_rules):
        if matches_whitelist_rule(operator, account, operation, rule):
            return (ALLOW, f"白名单规则 {rule.id} 通过", None)
    
    for exception in get_exceptions():
        if matches_exception(operator, account, operation, exception):
            if is_valid_timeframe(exception):
                return (ALLOW, f"特例 {exception.id} 通过", None)
            else:
                log_expired_exception(exception)
    
    for rule in sort_by_priority(relevant_rules):
        if matches_blacklist_rule(operator, account, operation, rule):
            if rule.action == "trigger_approval":
                approval_config = rule.approval_config
                return (NEED_APPROVAL, f"黑名单规则 {rule.id} 触发审批", approval_config)
            elif rule.action == "reject":
                return (REJECT, f"黑名单规则 {rule.id} 拒绝", None)
    
    # Step 3: 无规则匹配 = 允许（默认开放）
    return (ALLOW, "无相关规则，默认允许", None)


def execute_operation(operator, account, operation):
    """
    执行操作的完整流程
    """
    
    # Step 1: 评估策略
    result, reason, approval_config = evaluate_access_policy(operator, account, operation)
    
    # Step 2: 根据评估结果处理
    if result == ALLOW:
        # 直接执行
        execute_real_operation(operator, account, operation)
        log_operation(operator, account, operation, status="executed", reason=reason)
        return {"status": "success", "message": reason}
    
    elif result == NEED_APPROVAL:
        # 冻结操作，发起审批流
        approval_request = create_approval_request(
            operator=operator,
            account=account,
            operation=operation,
            reason=reason,
            approvers=approval_config["approvers"],
            timeout=approval_config["timeout_hours"]
        )
        notify_approvers(approval_request)
        log_operation(operator, account, operation, status="pending_approval", reason=reason)
        
        # 设置超时任务
        schedule_approval_timeout(approval_request)
        
        return {"status": "pending", "message": f"待审批，审批ID: {approval_request.id}"}
    
    elif result == REJECT:
        # 拒绝操作
        log_operation(operator, account, operation, status="rejected", reason=reason)
        alert_security_team(operator, account, operation, reason)
        return {"status": "rejected", "message": reason}


def handle_approval_result(approval_request, approval_result):
    """
    处理审批结果
    """
    
    if approval_result == "approved":
        # 审批通过，执行操作
        operator = approval_request.operator
        account = approval_request.account
        operation = approval_request.operation
        
        execute_real_operation(operator, account, operation)
        log_operation(
            operator, account, operation, 
            status="executed_after_approval",
            approval_id=approval_request.id,
            approved_by=approval_result.approver
        )
        notify_operator(operator, "审批通过，操作已执行")
    
    elif approval_result == "rejected":
        # 审批拒绝
        log_operation(
            approval_request.operator, 
            approval_request.account, 
            approval_request.operation,
            status="approval_rejected",
            rejection_reason=approval_result.reason,
            rejected_by=approval_result.approver
        )
        notify_operator(approval_request.operator, f"审批拒绝：{approval_result.reason}")
```

### 1.5 规则定义的最佳实践

#### 最佳实践1：规则应该原子化（Atomic）
**错误做法**：
```
规则: 金融主管在工作时间内可以处理退款
问题: "工作时间"定义模糊，"退款"包括哪些子操作？
```

**正确做法**：
```
规则1: 金融主管 + 操作类型=部分退款 + 金额<=1000元 → 直接通过
规则2: 金融主管 + 操作类型=全额退款 + 金额>1000元 → BOSS审批
规则3: 金融主管 + 操作时间=工作时间(09:00-18:00) → 放行
规则4: 金融主管 + 操作时间=非工作时间 → 发送实时告警
```

#### 最佳实践2：规则应该可度量（Measurable）
**错误做法**：
```
规则：禁止异常操作
```

**正确做法**：
```
规则：同一员工在1小时内操作超过100个商品 → 触发审批
规则：同一账号24小时内被多个IP访问 → 发送安全告警
规则：员工删除的商品价值超过1万元 → BOSS审批确认
```

#### 最佳实践3：规则应该有版本管理和灰度
```
规则发布流程：
草稿 → 灰度测试(10%用户) → 灰度扩大(50%用户) → 全量发布 → 监控 → 回滚(if异常)

变更记录：
版本1（2026-04-10）: 删除商品需BOSS审批
  变更人：CEO
  变更原因：误删事件
  影响范围：所有账号
  
版本2（2026-04-15）: 金融主管可直接删除库存过期商品（灰度50%）
  变更人：运营主管
  变更原因：加快库存周期
  风险评估：低（仅限过期商品）
  灰度计划：第1周50%，第2周100%
```

## 模块2：安全监管（Security Oversight）

### 2.1 功能定位
**本质**：实时监控员工操作，在风险行为发生的当下进行干预和记录。

**三层防御体系**：
```
事前拦截 (访问策略)           事中监管 (安全监管)          事后追溯 (操作审计)
├─ 规则定义                  ├─ 实时告警                  ├─ 日志查询
├─ 权限边界清晰              ├─ 操作冻结/隔离             ├─ 视频回放
└─ 高风险操作冻结            └─ 人工审查                  └─ 数据对比分析
```

---

### 2.2 安全监管的业务场景

#### 场景1：异常操作实时告警
**监控指标**：
```
指标1: 短时间内批量删除商品
  触发条件: 同一员工在1小时内删除商品数 > 50
  告警等级: 中
  响应: 发送告警给BOSS + 安全团队
  处置选项: [冻结账号] [实时查看] [忽略此次告警]

指标2: 异常地区登录
  触发条件: 员工在24小时内从IP地址不同的国家/地区登录
  告警等级: 高
  响应: 发送告警 + 发送员工确认邮件 + 启动二次认证
  处置选项: [冻结账号] [确认登录安全] [启用硬件密钥]

指标3: 数据导出异常
  触发条件: 员工导出订单数据超过1000条 OR 导出敏感数据（客户邮箱、支付信息）
  告警等级: 高
  响应: 自动拦截导出 + 发送告警
  处置选项: [重新提交申请] [BOSS审批] [取消操作]
```

#### 场景2：风险操作的实时冻结与隔离
**工作流程**：

### 2.3 安全监管的数据架构

#### 告警规则引擎
```python
class SecurityAlertEngine:
    """
    实时监控异常操作，生成告警
    """
    
    def __init__(self):
        self.alert_rules = {}
        self.alert_queue = Queue()
    
    def register_alert_rule(self, rule_name, condition_func, alert_level, action_func):
        """
        注册告警规则
        rule_name: 规则名称（如："批量删除告警"）
        condition_func: 条件函数，返回 True/False
        alert_level: 告警等级（Low/Medium/High/Critical）
        action_func: 告警触发时执行的函数（如：发邮件、短信、钉钉等）
        """
        self.alert_rules[rule_name] = {
            "condition": condition_func,
            "level": alert_level,
            "action": action_func
        }
    
    def evaluate_operation(self, operator, account, operation):
        """
        实时评估操作，生成告警
        """
        triggered_alerts = []
        
        for rule_name, rule_config in self.alert_rules.items():
            if rule_config["condition"](operator, account, operation):
                alert = {
                    "rule_name": rule_name,
                    "level": rule_config["level"],
                    "timestamp": now(),
                    "operator": operator,
                    "account": account,
                    "operation": operation
                }
                triggered_alerts.append(alert)
                
                # 异步执行告警动作
                self.alert_queue.put((rule_name, rule_config["action"], alert))
        
        return triggered_alerts
    
    def process_alert_queue(self):
        """
        后台处理告警队列（异步）
        """
        while True:
            rule_name, action_func, alert = self.alert_queue.get()
            try:
                action_func(alert)
            except Exception as e:
                log_error(f"告警处理失败: {e}")

    condition_func=lambda op, acc, act: (
        act.operation_type == "export_data" and
        (count_exported_rows > 1000 or contains_pii(act.export_data))
    ),
    alert_level="Critical",
    action_func=lambda alert: (
        freeze_operation(alert),
        notify_boss_with_approval_link(alert),
        audit_log(alert)
    )
)
```

#### 实时监控仪表板数据源
```
数据流转：

操作发生 → 操作日志存储(MQ异步) → 评估告警规则(实时) → 
告警生成(Redis缓存) → 前端仪表板查询 → 实时更新(WebSocket推送)

关键指标计算（基于时间窗口）：
├─ 过去1小时内: 高风险操作数 (Sliding Window)
├─ 过去24小时内: 各员工的异常操作排行 (TopN + Aggregation)
├─ 过去7天内: 账号的风险趋势 (Time Series)
└─ 实时: 在线员工的即时操作状态 (Stream Processing)

计算引擎：
Kafka → Storm/Flink实时计算 → Redis缓存 → API → 前端
```

### 2.4 安全监管的处置流程

#### 处置决策树
```
告警生成
├─ Low级告警
│  └─ 自动记录到日志，仅在仪表板展示，不主动通知
│
├─ Medium级告警
│  ├─ 邮件通知BOSS
│  ├─ BOSS在1小时内可以查看详情和处置
│  └─ 1小时无回应 → 升级为High级告警
│
├─ High级告警
│  ├─ 邮件+短信通知BOSS和安全主管
│  ├─ 可选：自动拦截敏感操作（不执行，冻结等待批准）
│  ├─ BOSS在15分钟内需确认
│  └─ 15分钟无回应 → 自动执行预设策略（通常是拒绝操作）
│
└─ Critical级告警
   ├─ 邮件+短信+钉钉群通知BOSS、安全主管、CEO
   ├─ 自动冻结账号（防止进一步损失）
   ├─ 立即启动紧急人工审查
   ├─ 可选：强制下线操作员（需CEO确认）
   └─ 详细记录所有处置步骤（审计链路）
```

#### 处置动作（Action）
```
BOSS在看到告警后可执行的动作：

1. [允许操作] - 确认此操作无问题，继续执行
   记录: "BOSS在14:50允许李四删除商品SKU_12345"
   
2. [拒绝操作] - 取消该操作
   记录: "BOSS在14:50拒绝李四的删除操作，原因：热销商品不应删除"
   
3. [请求信息] - 要求操作者补充更多上下文
   记录: "BOSS请求李四解释删除原因"
   流程: 冻结操作 → 李四提交说明 → BOSS再次评估
   
4. [冻结账号] - 暂停该账号的所有操作（紧急措施）
   记录: "BOSS在14:50冻结Amazon账号A，原因：发现异常删除操作"
   通知: 李四收到通知（你的账号已被冻结，请联系BOSS）
   
5. [启动审查] - 调出该操作的完整视频和日志，进行深度分析
   记录: "安全主管开始审查"
   流程: 查看视频 → 查看日志 → 查看相关的其他操作 → 给出结论
   
6. [提升事件] - 将此告警升级为"安全事件"，启动跨部门调查
   记录: "CEO将此事件升级为P1级安全事件"
   流程: 邮件通知全体管理层 → 安全应急会议 → 根因分析 → 改进计划
```

### 2.5 安全监管的技术实现

#### 告警执行的完整流程代码
```python
class SecurityOversightSystem:
    """
    安全监管系统的核心逻辑
    """
    
    def on_operation_executed(self, operator, account, operation):
        """
        操作执行时触发
        """
        
        # Step 1: 评估告警规则
        triggered_alerts = self.alert_engine.evaluate_operation(operator, account, operation)
        
        # Step 2: 若有告警，开始处置流程
        for alert in triggered_alerts:
            self.handle_alert(alert)
        
        # Step 3: 记录操作和告警到审计日志
        self.audit_log.record({
            "operator": operator,
            "account": account,
            "operation": operation,
            "alerts_triggered": len(triggered_alerts),
            "timestamp": now()
        })
    
    def handle_alert(self, alert):
        """
        处置一条告警
        """
        
        # Step 1: 保存告警到数据库
        alert_id = self.db.insert_alert({
            "rule_name": alert["rule_name"],
            "level": alert["level"],
            "operator": alert["operator"],
            "account": alert["account"],
            "operation": alert["operation"],
            "created_at": now(),
            "status": "pending"  # pending / resolved / ignored / escalated
        })
        
        # Step 2: 根据告警等级执行相应的动作
        if alert["level"] == "Low":
            self.log_to_dashboard(alert)
        
        elif alert["level"] == "Medium":
            self.send_email_notification(
                recipients=self.get_boss_emails(),
                subject=f"安全告警: {alert['rule_name']}",
                body=self.format_alert_message(alert),
                action_link=self.generate_alert_action_link(alert_id)
            )
            
            # 启动1小时计时器
            self.schedule_escalation(
                alert_id=alert_id,
                escalate_to_level="High",
                after_seconds=3600
            )
        
        elif alert["level"] == "High":
            self.send_sms_notification(
                recipients=self.get_boss_phones() + self.get_security_manager_phones(),
                message=f"高危告警: {alert['rule_name']}，请立即查看: {self.generate_alert_link(alert_id)}"
            )
            self.send_dingtalk_notification(
                group_id="boss_security_group",
                message=f"🚨 高危告警\\n{self.format_alert_message(alert)}"
            )
            
            # 可选：自动冻结操作（取决于规则配置）
            if self.should_freeze_operation(alert):
                self.freeze_operation(alert["operation"])
            
            # 启动15分钟计时器
            self.schedule_escalation(
                alert_id=alert_id,
                escalate_to_level="Critical",
                after_seconds=900
            )
        
        elif alert["level"] == "Critical":
            self.send_email_notification(
                recipients=self.get_all_senior_managers_emails(),
                subject=f"🚨 严重安全事件: {alert['rule_name']}",
                body=self.format_critical_alert_message(alert)
            )
            self.send_sms_notification(
                recipients=self.get_ceo_phone(),
                message=f"严重安全事件，已自动冻结账号: {alert['account']}"
            )
            
            # 自动冻结账号
            self.freeze_account(alert["account"])
            
            # 启动人工审查
            self.create_incident_ticket(
                alert_id=alert_id,
                priority="P1",
                assigned_to="security_team"
            )
    
    def escalate_alert(self, alert_id):
        """
        告警升级（当前级别无人处置时调用）
        """
        
        alert = self.db.get_alert(alert_id)
        current_level = alert["level"]
        
        if current_level == "Medium":
            # 升级为 High
            self.db.update_alert(alert_id, {"level": "High"})
            self.handle_alert_escalation(alert, new_level="High")
        
        elif current_level == "High":
            # 升级为 Critical
            self.db.update_alert(alert_id, {"level": "Critical"})
            self.handle_alert_escalation(alert, new_level="Critical")
    
    def boss_resolve_alert(self, alert_id, action, reason=None):
        """
        BOSS处置告警
        """
        
        alert = self.db.get_alert(alert_id)
        
        if action == "approve":
            # 批准操作继续执行
            self.db.update_alert(alert_id, {
                "status": "resolved",
                "resolved_by": get_current_boss(),
                "resolved_action": "approve",
                "resolved_at": now()
            })
            self.unfreeze_operation_if_frozen(alert["operation"])
            self.log_operation_as_approved(alert["operation"])
        
        elif action == "reject":
            # 拒绝操作
            self.db.update_alert(alert_id, {
                "status": "resolved",
                "resolved_by": get_current_boss(),
                "resolved_action": "reject",
                "resolved_reason": reason,
                "resolved_at": now()
            })
            self.cancel_operation(alert["operation"])
            self.notify_operator(
                alert["operator"],
                f"您的操作已被BOSS拒绝，原因: {reason}"
            )
        
        elif action == "investigate":
            # 启动深度调查
            self.create_security_investigation(alert_id)
            self.db.update_alert(alert_id, {"status": "investigating"})
        
        # 记录BOSS的处置动作
        self.audit_log.record({
            "event_type": "boss_alert_resolution",
            "alert_id": alert_id,
            "boss": get_current_boss(),
            "action": action,
            "reason": reason,
            "timestamp": now()
        })
```

### 2.6 安全监管的风险点

#### 风险1：过度告警导致"告警疲劳"
**问题**：如果告警规则设置太敏感，会导致大量误报，BOSS最后干脆忽视所有告警。

**缓解方案**：
```
1. 规则调优：收集4周告警数据，计算每条规则的精确率和召回率
   ├─ 精确率 < 60% → 调松规则阈值
   ├─ 精确率 > 95% → 可以调严
   └─ 目标：精确率 80%+ 的平衡

2. 规则分级：
   ├─ 核心规则（严格）：必须通知BOSS
   ├─ 普通规则（中等）：邮件通知
   └─ 参考规则（松散）：仅记录到仪表板

3. 智能告警：
   ├─ 基于历史模式学习（这个员工通常每小时删除50个商品）
   ├─ 如果这次是200个 → 高度异常 → 告警
   ├─ 如果这次是60个 → 轻微异常 → 降级处理
   └─ 使用异常检测算法（如 Isolation Forest）
```

#### 风险2：告警处置的人工延迟
**问题**：BOSS可能不在线或响应缓慢，导致高风险操作没有及时止损。

**缓解方案**：
```
1. 自动化处置：
   ├─ 部分告警可配置自动拒绝（如：删除最高价值商品）
   ├─ 可配置自动冻结账号（如：检测到账号劫持）
   └─ BOSS可在"告警规则"中预设自动处置策略

2. 升级机制：
   ├─ 告警15分钟无回应 → 自动升级到CEO
   ├─ 自动发送CEO的个人手机短信
   └─ 必须有人工确认（不能自动化到底）

3. 值班制度：
   ├─ 24小时值班BOSS轮换
   ├─ 高峰期（晚上8点-凌晨1点）需要值班人员
   └─ 值班信息公开，员工知道谁在线
```
