## 💡 个人思考（最高优先级）

### 1. "rag 词典" 和 "url_resolver 查注册系统" 不是一件事 —— 产品侧的流水线认知

今天犯了一个典型的"功能混淆"误判：看到 `url_resolver.py` 能查 dev 资源池 + 补登缺失 URL，就以为可以丢掉 `rag_*.json`。被我自己挡回来的时候才意识到，**这是流水线上的两个前后步骤，不是同一件事的两种实现**。

- **rag**：输入**中文描述**（"买家消息页"），输出 **URL pattern**（`https://*.amazon.*/messaging/inbox`）—— 是 SKILL **产出字段的推理**阶段
- **url_resolver**：输入 **URL pattern**，输出 **url_id** —— 是策略**写入前的 id 解析**阶段

两者功能定位完全不同：
- rag 是**业务 URL 词典**（带中文描述 + 别名 + 关键词的语义索引）
- dev 资源池是**已注册 URL 的登记表**（只有 name 和 url_pattern，没有语义字段）

> **教训**：在做"A 能不能替代 B"的判断时，先问"A 的输入 == B 的输入吗？A 的输出 == B 的输出吗？"。输入输出不匹配就不能替代，不管功能看起来多相像。这次要不是我在回答前强行拆开流水线图（"用户 → SKILL 推理 → form_output → api_连调 翻译 → 落库"），我会真的在打包时误删 rag，事后还不知道为啥 demo 跑挂。

### 2. SKILL 是 prompt 不是代码 —— 架构约束必须讲清楚

用户提出 "改 SKILL.md Phase 0 读真接口" 时，我第一反应是按字面理解做。但想清楚后发现：**SKILL 本质是对话指令文档，Read 工具读文件是它能做的事，但 Read 工具不能"调 API"**。

想让 SKILL 调 API，必须绕：
- 方案 A：SKILL.md 在 Phase 0 指令 "Bash 跑 `python -m client.snapshot xxx`"，Bash 工具跑脚本调 API 把结果写到 `cache/xxx.json`，再 Read 回 SKILL 上下文
- 方案 B：让 SKILL 在对话中插入 API 调用（Claude Agent 模式，复杂度指数级上升）

实际可用方案只有 A，且**必须有 Claude Code 级别的工具环境**（Bash 工具可用）。如果未来这个 SKILL 要部署到"只有对话、没有工具"的平台（比如飞书 aily），A 方案直接崩 —— 那就必须回退到"预先同步数据到文件再 Read"。

> **教训**：PM 给出技术方向时（比如"让 X 改成动态调"），自己先在脑子里走一遍"X 是什么载体 / 它的执行能力边界 / 最简可行路径是什么"。不要让开发/AI 按字面猜你要什么，否则得到的要么是过度工程（花 1~2 天自己实现一个 Agent 架构），要么是机械理解（真改 prompt 文字但实际跑不起来）。这次及时发现并提出"折中方案：改 prompt 文字 + Bash 调 snapshot 脚本"，避免了一次无用的大 refactor。

### 3. plan → 批准 → 执行 → 交付，一天压 7.5 天的工作是可行的

原 plan（05-11 批准）估计阶段 A 2 天、阶段 B 3 天、阶段 C 1.5 天、阶段 D 1 天共 7.5 天。今天一天基本把 A + B + SKILL 集成 + 打包都做完了，**压缩 4 倍**。

复盘原因：
1. **plan 写得足够细**：目录结构、模块职责、12 条映射规则都预置在 plan 里，执行时没有"还要想怎么做"的空档
2. **Swagger 接口文档是权威**：每个字段问"是什么/必填吗/枚举值"都能 Ctrl+F 查到答案，没有回溯踩坑
3. **"纯函数翻译层 + 反查外置"的分层**：`route2_builder.py` 可单元测试（15 个 case 都过），上游 `organization.py/url_resolver.py` 单独验证，E2E 跑的时候各层问题隔离开了
4. **Claude Code 作为 IDE**：可以同时改 5 个文件、grep 全库、跑 Bash —— 省掉频繁切工具的时间
5. **今天 dev 鉴权开关"碰巧被关"**：5/12 早上验证 dev 依然拒 session；下午"明文可以请求了"直接把端到端全跑通

风险面：
- 如果今天 dev 始终拒 session，阶段 B 的端到端验证做不了（只剩离线单测），后续打包就只能做"死包"（用户 demo 时还得回来跑真链路）
- 这种 "dev 环境状态决定 PM 工作进度" 的依赖很不健康，但作为实习生没权限改变

> **洞察**：工程侧加速的顶能是"让执行阶段没有思考，所有决策前置到 plan"。PM 负责**把模糊需求拆成清单**，执行效率完全在清单的粒度上。今天证明：一份写清楚的 plan + 一个权威接口文档 + 一个封闭的 IDE 环境，是 PM 侧最接近"工程师"的模式。

### 4. demo 打包的 3 版迭代 —— "每删一份都要讨论"的价值

第 1 版 220KB（含 api_实测 3 份核心文档）→ 第 2 版 177KB（去 api_实测）→ 第 3 版 163KB（去 templates + 对话记录样本 + example_output）。每一版用户都问了"这个能不能删"，每一版我都被迫重新盘点这份文件的具体作用，结果发现：
- api_实测 看起来是"开发过程副产物"，其实**工作原理详解.md 里有 3 处硬链接**，直接删会留死链
- templates 看起来是"模板库，SKILL 需要"，其实**联调模式下走 cache/templates.json**，assets 里是老版本不用
- 对话记录 3 份看起来是"SKILL 要引用的示例"，其实**SKILL 只 Write 到这个目录不 Read**，3 份历史样本完全是过去产物
- rag 看起来是"知识库，有接口替代了就不用"，其实是**唯一的语义索引**，dev 接口替代不了

> **教训**：用"默认保留"的思路打包，会带很多不必要的文件；用"默认删除"的思路盘点，又会误删核心资产。**正确做法是"每个文件回答三个问题"**：
> 1. runtime 阶段会 Read 吗？（决定能否删）
> 2. 用户打开后会点吗？（决定会不会看到死链）
> 3. 被这份文件引用的其他文件会留吗？（决定引用链是否断裂）
> 用这三个问题过一遍，删文件的决策就不是"感觉没用"而是"经过证伪"。

### 5. dev 环境不稳的应对 —— "session 开关不是常态，更像联调闸门"

5/12 早上踩坑：dev 拒 session，我以为环境变了 / 鉴权机制升级了。讨论到下午才发现：**dev session 校验是一个开关**，联调时会被开发临时关掉，联调结束又开上。我们 5/9 跑通时恰好是关的，5/12 早上是开的，下午又关了。

这个认知改变了我对 dev 环境的心智模型：
- 以前：dev = 一个"和生产一模一样但明文"的环境
- 现在：dev = 一个"和生产共享数据但可临时绕过鉴权"的环境

→ 对 PM 来说，**遇到 dev 10001 错误时的第一反应不该是"我做错了"，而是"先问开发 session 开关"**。这能省 2 小时的尝试。

> **洞察**：和第六周"dev 环境路径前缀"（`/api/v5/` 去 v5）一样，dev 的具体行为细节**只能靠实测 + 问人**，不能从 Swagger / 官方文档推断。我得建立一份 "dev 环境行为手册"（session 校验开关 / 路径前缀规律 / URL 资源池污染 / 角色数据稳定性 / ret 码含义 / ...），和 "接口 schema" 不一样，这是**运维状态文档**。

### 新增方法论

- [**第 18 条：功能替代判断的输入输出对齐法**](../产品方法论/产品思维框架.md#18)（新增）—— 判断 A 能否替代 B，先看输入类型、输出类型、语义字段匹配度。rag 和 url_resolver 就是反例：输入输出完全不同却看起来相像。
