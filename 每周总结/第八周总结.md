## 💡 个人思考（最高优先级）

### 1. AI SDLC 是认真在帮 PM 兜底，不是花架子

之前一直把 AI SDLC（spec-product-clarify / spec-product-prd 等系列 skill）当流程仪式 —— 觉得"我自己写 PRD 也能写出来啊，何必走这一套"。今天一整天走完才意识到：**这套流程的真正价值不是"让 AI 帮我写 PRD"，而是"让 AI 用结构化协议替我兜住所有可能漏掉的不确定性"**。

具体兜在三处：
- **澄清阶段一次只问一题**：我催"一次问完吧"，AI 仍按规矩问 1 题，剩下转 V 项 — 这不是 AI 在"反抗"，是它知道高杠杆题答完会消解低杠杆题，一次问完反而浪费我的低杠杆决策
- **CONTEXT GAP 不脑补**：design.md 里 5 处 "应引用 X，但 X 不存在" 显式标出来，没有用一段假大空的"对齐声明"糊过去 — 后续 D1 阶段做 project-discover 时这些 GAP 就是触发动作清单
- **NEEDS CLARIFICATION 阻断 I2**：plan.md 完成后 ROUTER_SUMMARY blocked=true，不让我直接进 spec-execute；逼我先决策 C1（工作区位置）才能解锁 T1 — 如果不阻断，我可能等到 T7（写回兜底）写一半才发现 V-017 后端不愿做，半段代码白写

> **教训**：流程仪式的价值要看"它阻止了什么"，不是"它产出了什么"。AI SDLC 阻止的是 PM 在压力下跳步、脑补、把不确定性写成确定性。
> 这周犯过两次"想跳步"：① 想跳 R3 直接进 D1（被 R3 提示路径 A1/A2/B 三段交互"需要原型"挡回） ② 想跳 D0 强制门禁直接 I1（using-aisdlc Router 硬规则"R→I 必经 D0"挡回）。每一次被挡都意外发现了原本会漏掉的东西。

### 2. 在 AC 里写硬时延数字是 PM 给自己挖的坑

下午写 PRD 时随手写了 AC-003 "用户回答后 SKILL 在 ≤2s 内生成 JSON"，被用户当场质疑："紫鸟 LLM 网关无 prompt cache、走 AWS Bedrock，2s p95 几乎不可能"。这条其实是从 raw §4 "执行 6 Phase 工作流" 一字面带过来的，我没想过 6 Phase 串行 + 每次重传 prompt 的真实时延。

被质疑后做了两件事：
1. 全 spec 审计，把所有时延类硬阈值（"p95 ≤2s"、"≤2s 启动"等）转成 V 项，让 PM/UX 据真实压测数据回填
2. 在 PRD §5 加规则-11 "性能指标先压测后写死"作为**总原则**，未来同类项目可复用
3. 新增 V-012 LLM 端到端时延 — 这是常见漏掉的维度（很多 PRD 只写接口拉取时延，忘了 LLM 自身的时延）

但还是要区分清楚 —— 不是所有数字都软化：
- **业务规则**保留硬数字：重试 2 次 / 间隔 ≥1s / 路径 B ≤6 轮 / 保留期 90/180/365 天 → 这是产品决策，不需压测
- **性能指标**软化进 V：JSON 生成时延 / 接口启动 p95 / 首字反馈 → 需要数据

> **洞察**：B 端产品 PRD 里的"性能要好"是没用的，但写"≤2s"也是没用的（被当成上线门槛但根本测不达标）。**把性能指标和验证清单绑定**：每条性能指标都附压测方法 + 触发动作（不达标怎么降级），上线前回填具体数字 → AC 才有意义。这条规则做总原则比做某条 AC 更有价值，因为同事下次写新 PRD 不会再犯。

### 3. 让 demo 当 evidence，不当 R4 走查替代品

我本来打算按 AI SDLC 标准流程做 R4（生成 Demo 工程供干系人评审）。用户提到"我已经在本地跑通了流程"时，我先以为是 demo2 v3.0，实际看 reports/ 目录后才意识到：**这是 5/14 完成的真实端到端联调（用户原话→AI→真实平台落库 rule_id=16814645350935）**，远超 R4 走查的目的。

如果我把这份 demo 当成 R4 走查的低保真原型，相当于把已经"上路实测过的整车"降级成"3D 模型供你转一圈看"。正确做法是：
- 跳过 R4（独立做低保真 demo 反而是降级）
- 把 demo 转化为 evidence —— `evidence/reports/` 拷贝 5 份联调报告 + `evidence/fixtures/` 拷贝 11 个 JSON 样本 + 写 README 索引（证据↔V 项/AC 映射）
- 后续 design.md / plan.md 里所有"D6 静默丢弃自检 / D7 URL 防撞 / 已知坑位 P1-P7"都从 evidence 里取证据 — 这些不再是"假设"而是"实证"
- 实施工程主体（client/translator/runner...）作为 I2 工作区基线（C1=A 迁入本 repo `code/access-policy-skill/`）

> **教训**：**用户给你的资产先看实质再决定流程怎么走**，不要按 SOP 流程的"应该到了 R4 阶段就做 R4"机械执行。流程是工具，不是目标。如果实证级 demo 已存在，转化为 evidence 比生成走查产物对项目价值大得多。

### 4. NC 阻断不是"拒绝推进"，是"分级求救"

I1 做完 plan.md 时，我标了 5 项 NEEDS CLARIFICATION（C1-C5），把 ROUTER_SUMMARY 写成 `blocked=true`。第一感觉是"卡住了，进度推不下去"，但实际作用恰恰相反 —— 这是把"我作为实习生没决策权"翻译成"我需要谁来回答什么"的工具：

| NC | 谁来答 | 截止 | 不答会怎样 |
|---|---|---|---|
| C1 工作区位置 | PM 自己 | 5/19 | T1-T17 全卡住 |
| C2 SKILL 接入文档 | AI 工作台架构 Owner | 5/22 | T2 卡住，全 LLM 联调延后 |
| C3 V-017 临时缓存 | 访问策略后端 Owner | 5/22 | T7 写回兜底卡住 |
| C4 V-005 审计签字 | 安全合规 | 5/26 | 不签字不能上线 |
| C5 V-006 字段白名单 | 产品经理 | 5/27 | 路径 B 字段没法定义 |

每条 NC 都附三段：缺什么 → 怎么取证 → 信号 → 触发动作。用户看到这五条时不需要再问"你卡在哪"，直接知道"今天发四条飞书消息给四个不同 Owner"。**这是 AI 替我把"我不知道"翻译成可分配的对外行动项**。

> **洞察**：实习生的核心瓶颈是"决策权 + 信息不对称"，不是"做事能力"。AI SDLC 的 NEEDS CLARIFICATION 协议正好对症 — 把决策权外推（让 AI 帮我列出"由谁决策"），把信息缺口结构化（缺什么 + 怎么取证）。这条比任何"我亲自跑去问"的工作模式都高效，因为它把信息收敛进 plan.md 这个 SSOT，而不是留在我的微信记录里。

### 5. spec ↔ code ↔ evidence 同仓的代价是值得付的

C1 NEEDS CLARIFICATION 让我选工作区位置：A 迁入本 repo / B 仓外引用 / C 独立仓库。我推荐 A 但也列出代价（迁移成本、仓库定位变化）。用户秒选 A。

迁完发现 A 真的对：
- spec 里的相对路径 `evidence/reports/...` 不会断链（B 方案要写 `../../访问策略SKILL_fromLPP/...`，换电脑就死）
- code 改动可以和 spec 同分支 PR（B/C 都得跨仓协调）
- commit history 按 SDLC 阶段拆 6 个 → `git log --oneline` 直接看到流程时间线（chore/feat-R0-R3/docs-evidence/feat-D1-D2/feat-I1/feat-I2-T1）

但有两个细节差点踩坑：
1. **敏感文件**：源工程 `config/session.dev.json`（含 oauth_string）+ `config/env.dev.json`（含内网 IP）必须排除。我用 PowerShell 递归复制时显式排除这两个文件 + logs/ + __pycache__/ + .venv/，并在迁移后用 `grep -rl "oauth_string"` 防御扫描确认无残留
2. **不用 Move 用 Copy**：用户先要我 Move，但 Copy 保留源目录作回滚更安全 — 万一迁移有问题不影响仓外 demo 工程继续可用

> **洞察**：**用 git 仓的方式就是用流程的方式**。同仓 = 同流程；分仓 = 协调跨流程。实习生最缺的就是"协调多流程"的能力（你不知道怎么催不同团队的 PR），所以默认选同仓最稳。代价（仓变重、迁移成本）相对实习节奏来说都是小成本。
