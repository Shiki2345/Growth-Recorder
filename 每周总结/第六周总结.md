## 💡 个人思考（最高优先级）

### 1. "AI 学会 vs AI 被证明学会"是两个量级的问题

本周最大的认知升级：这两件事不是时间差，是**证据结构的差**。

- **AI 学会**（直觉判断）= 看几个案例跑对了，就信它有能力
- **AI 被证明学会**（工程判断）= 覆盖率 × 多样性 × 一致性 × 抗干扰，四层证据同时成立

今天的测试过程里，我对自己先做了一次"从直觉判断升级到工程判断"的拉升。最初觉得"Layer 2 22 条全命中就够了"，但仔细拆能力边界才意识到：
- 22/57 的覆盖率只能支撑"**主干能力**"陈述
- Layer 3 的 93.4% 一致性才能支撑"**稳定**"陈述
- 红线触发 0 次才能支撑"**不出事**"陈述
- 三者缺一，都不能对老板说"AI 学会了"

> **教训**：以后汇报 AI 能力时，默认先把"能力"拆成"覆盖率 + 稳定性 + 抗干扰 + 红线"四个维度分别论证；只要有一个维度没数据，就不说"学会了"，说"MVP 主干验证通过"。这比"能力跑通"更准确，也更经得起追问。

### 2. LLM 硬约束必须靠"清单 + 禁止词 + 固定模板"，不是"原则描述"

今天经历了 4 轮 SKILL.md 加固，最深的体会：**LLM 对不同强度约束的响应率差一个量级**。

- 第 1 轮写"应该匹配模板" → AI 偷懒不 Read 模板文件
- 第 2 轮写"必须匹配并宣告结果" → AI 说"我先理清几个参数"省略宣告
- 第 3 轮写固定话术"第一句必须按此句式开头：我匹配到《X》..." → 命中时稳定开场，但短模糊需求下 AI 仍滑到 RBAC
- 第 4 轮写"禁止词清单 + 合法词清单对照表" → 红线守住

每次失败的共同点：**上一级约束没足够的"违反成本"**。AI 会选择最省力的理解方式，只有把违反成本明确标出来（"出现这种开场白 = 违规"、"❌ 禁止出现的词：XXX"），它才会当硬纪律执行。

> 这推翻了我原来的假设——"好好写原则，AI 能理解"。**AI 能理解原则，但不会主动对齐到最严格的原则**，除非你把"最严格"显式化。

### 3. 输出锚定是对抗"训练分布引力"的唯一手段

早上 D3 测试出现严重脱轨：用户说"财务的权限"，AI 输出了 ERP 总账 / 银企直连 / HR 薪酬 / CRM / 代码仓库的权限矩阵——**完全脱离紫鸟浏览器的概念体系**。

根因诊断：LLM 训练语料里，"财务权限"的最常见搭配就是企业 ERP/HR/财务系统。用户输入没提"紫鸟"或"Amazon"等锚点词时，AI 默认回到这个分布最浓的区域。

解法：在 SKILL.md 显式加"输出锚定"红线，列两份对照清单——
- ❌ 禁止出现的跨域词（ERP/总账/银企直连/HR/CRM/代码仓库/设计稿...）
- ✅ 合法的本域词（紫鸟 URL/Amazon Seller Central 模块/TikTok Shop 模块...）

测试验证：加固后零触发。

> **这是 AI 产品设计里一个反直觉的结论**：你不能靠"告诉 AI 去做对的事"来保证对齐，必须"告诉 AI 不要去做哪些错的事"。正向约束（"必须"）引导有限，负向约束（"禁止"）才能对抗训练分布引力。

### 4. MVP 路径的关键：信任数据本身的语义信号

开工时面临一个选择：要不要给 62 个模板每条都补 `what / how / 匹配关键词 / 匹配别名`？原计划是补的——看起来更严谨。

决策过程：
- 补：~60 模板 × 10 分钟标注 = 10 小时成本
- 不补：直接用 `template_name + description + URL 反查 RAG` 三层组合打分

心里没底，所以先不补试一下。结果 Layer 2 22/22 命中。

> 复盘：**原来的语义信号已经够强了**。模板名是人写的（"财务人员访问限制"直接包含"财务""人员"），描述是人写的（"限制财务人员不能访问亚马逊其他无关页面"），这些加起来的"原生信号"已经能支撑子串打分到高置信阈值。前置补标注是 **YAGNI 典型**——你担心的情况没出现，标注就白做了。

**这个反思可以推广**：做任何"AI 理解层"的前置工作时，先问一个问题——**原数据里现有的字段已经提供了多少语义信号？**如果够，就不要补；如果不够，补最少的那条（比如只给关键 10% 的模板补，不是全量）。

### 6. "静默失败"是后端 API 世界里最贵的 bug（05-09 主线 2）

今天最扎的一个感受：从 13:00 提交第一条规则到 19:30 定位根因，中间花了 **6 小时**——不是 API 报错多难读，而是**它压根不报错**。

整个诊断链里，同一个现象（configs=[]）被我用四个完全不同的理论解释过：

| 轮次 | 我的假设 | 为什么信 | 怎么被推翻 |
|---|---|---|---|
| ① | `configs[].id` 是非法值 | 正常 detail 里 id 是 6 位数，我传的是 17 位 url_id，结构不一致 | 删掉 id 重提 → 仍然空 |
| ② | 黑白名单**模式级冲突**，dev 环境 24 条白名单"压制"了我的黑名单 | 返回的 24 条冲突列表很吓人，每条都标 `active_range=3`（白名单）| 扫 dev 全部规则，发现 156 条黑名单规则同环境共存，19/20 样本 configs 非空 |
| ③ | `preset_type` 是需要传的输入字段，传 `preset_type:1` 就能强写 | 对照组（cjytest）detail 里 preset_type=1 | 提交后 detail 查出来被**覆盖回 0**，不受 payload 控制 |
| ④ | 换用百度 url_id（其他规则在用的）→ configs 完整入库 ✅ | 诊断收敛了：**url_id 无效**是唯一真因 |

每一轮我都写了很自信的分析发给用户，每一轮都被实测推翻。这个过程让我明白了一件事：**静默失败 API 的诊断不能靠"看起来合理"，必须靠"能被实验证伪"**。

前三轮我都犯了同一个错——我在**解释现象**，不是在**设计实验**。解释现象的理论无限多（就像我连编三个），但设计实验的理论必须提出"如果 X 成立，那么 Y 应该发生；如果 Y 没发生，X 就证伪"。

后三步（_测试3/4/5）每一步都是**单变量对照**：只改一个字段，其他照抄上一条测试。所以当 _测试5 成功而 _测试3/4 失败时，root cause 就被机械定位到"换了的那个 url_id"上。**单变量对照不靠聪明，靠纪律**。

> **教训**：下次遇到"ret:0 但效果不符合预期"的静默失败，第一件事不是想"为什么"，是做三件机械动作：
> 1. `detail` 查一下字段到底是不是存进去了——list 能看到规则≠字段都存了
> 2. 把失败样本和成功样本并排字段对照，找出差异点
> 3. 每个差异点做**单变量对照实验**，一次只改一个

### 7. "敢于推翻自己"比"第一次就说对"更重要（元认知层）

今天前三轮诊断都错了，还都写了相当详细的分析（甚至写进了文档）。但当实测数据推翻假设时，我没有"尝试挽救原理论"，直接承认错、写进文档的"已证伪"章节、改结论。

这件事乍看像是小事，但对照一下：**很多真实协作里，人写完一份自信的诊断报告后，会下意识地"找证据支持自己、解释掉反例"**，而不是接受证据、改变结论。AI 做这件事比人容易一些（没有"面子成本"），但也不是天然就会——你得把"已证伪"写进文档、把 memory 里的旧断言删掉，这样下次新会话的自己才不会被误导。

今天的 workflow 文档里有一整段叫"⚠ configs 静默丢弃的根因：url_id 无效（2026-05-09 完整诊断）"，开头第一句话就是 "**不是黑白名单模式冲突**（之前一度以为是，已证伪）"。这句话是写给未来的我看的——防止下一次被旧理论带偏。

> **做法沉淀**：把被证伪的假设**显式写进文档**，标注"已证伪"、附带证据。不要只留下正确结论——因为未来的人（包括未来的自己）很可能会独立走一遍你走过的错路。**把错路钉死**，比只讲正路更有价值。

### 8. "前端 URL" 和 "后端真实路径" 可以差一层

今天花了最多无谓时间的一段：用户给我前端抓包的请求 URL `/api/v5/security/access_rule_url/add`，我对着直接打 dev 后端 → 返回 `{"raw":"no module"}`。

试了 10 分钟猜不同 prefix 才反应过来：**前端路径是经过代理层的版本化路径，dev 直连后端要去掉 `/v5/`**。确认以后，从 no module 变成 pydantic 的明文错误，瞬间解锁。

这件事有一个**可推广的规律**：

| 层 | 路径样式 | 特征 |
|---|---|---|
| 前端/代理层（抓包里看到的）| `/api/v5/...` / `/api/v1/...` | 带版本前缀，可能经过统一 gateway |
| 后端直连（dev 裸调） | `/api/...`（去掉 v*）| 模块名保留，版本去掉 |

下次再有新模块要探：看前端抓包的 URL，去掉版本号段，直接 POST 一个空 body 看 pydantic 错——如果能报字段 missing，说明模块对上了。这个从"摸黑猜"到"有法可循"的转换，值得沉淀成方法论（见下面方法论 17）。

### 9. 跨环境 ID 污染是隐蔽的 Bug 模式

今天的诊断回头看，最底层的错误其实发生在**最开始**——我用了**生产环境** hqyTest 抓包里的 `url_id=17237191297447`（mooncell），在 **dev 环境**里根本不存在。后面的 4 轮诊断只是在绕圈子验证这个前提错了。

这类 bug 有几个共同特征：
- **跨环境/跨租户**：把环境 A 的数据当作环境 B 的数据用
- **ID 无拼写错**：17 位数字看起来完全合法
- **后端无校验**：系统没检查"这个 id 在当前 scope 下是否存在"
- **报错静默**：整条 config 被丢掉，但顶层 ret 仍是 0

> **识别方法**：任何时候复用"从别处抓包得来的 ID"，都要先问一句——**这个 ID 是在当前环境注册的吗？** 如果不确定，调一次 list 或 detail 接口验证一下。几百毫秒的成本，避免几小时的诊断。

### 10. 未解的问题（05-09 延伸）

- 既然 dev 环境已经能全链路跑通，**生产环境的加密信封**要不要做？做的话信封里的 `machine_string` / `public_properties` 怎么生成？本地有没有能用的示例？

## 📚 方法论沉淀

### 方法论 11：LLM 硬约束的四级强度梯度（新增）

LLM 对不同强度约束的响应率差一个量级。设计 Skill 提示词时按以下梯度从弱到强上行，只在上一级失效时才升级：

| 级别 | 话术形式 | AI 执行率（经验值）| 适用 |
|---|---|---|---|
| 1 · 软提示 | "应该先..."、"建议..."、"可以考虑..." | ~30% | 友好建议，不强制 |
| 2 · 原则约束 | "必须先..."、"不得..."、"禁止..." | ~60% | 基础规则 |
| 3 · 固定话术 | "第一句必须按此句式开头：我匹配到《X》..." | ~90% | 关键输出 |
| 4 · 禁止清单 + 合法清单对照 | ❌ 禁止词：XXX / ✅ 合法词：YYY | 接近 100% | 红线级 |

**实战案例（本周 Phase 1.5 加固）**：
- 第 1 轮只用级别 2 "必须匹配模板" → AI 偷懒不 Read
- 升到级别 3 "第一句必须按此句式开头" → 命中时稳定，但短需求下仍脱轨
- 升到级别 4 "❌ 禁止 RBAC/ERP/HR/CRM 词 / ✅ 合法紫鸟模块词对照" → 红线守住

**选择原则**：不要一上来用最强约束，会牺牲 AI 的灵活性；每升一级都得有失败 case 证据。

### 方法论 12：测试的三层证据结构（新增）

验证 AI 能力不能只看"跑对几条"，要分三层独立论证：

| 层 | 测什么 | 数据产出 | 样本量 | 时间成本 |
|---|---|---|---|---|
| Layer 1 · 覆盖率 | 算法覆盖的范围边界 | "我测过 N% 的模板" | 全量 | 可脚本化 |
| Layer 2 · 多样性 | 每个类别都能 work | "所有类别抽样命中率 X%" | 每类 1-3 条 | 小时级 |
| Layer 3 · 一致性 | 同题多次跑稳定 | "关键题连跑 3 次一致率 Y%" | 关键题 × 3 次 | 小时级 |

**组合解释力**：
- 只有 Layer 1 没 Layer 3：覆盖率高但每次跑结果不同 → 不能产品化
- 只有 Layer 3 没 Layer 1：稳定但盲区大 → 尾部风险未知
- **Layer 2 是最佳性价比**：18-22 条覆盖多样性，小时级完成，能支持"主干能力验证"陈述

**本周实测**：Layer 2 22/22 命中 + Layer 3 93.4% 一致性 → 可以说"模板匹配主干能力验证通过"，但**不能**说"完全学会"（因为 Layer 1 覆盖率 38.6%）。

**适用场景**：反推实验、RAG 评估、Skill 行为验证、任何"AI 能不能做 X 类任务"的判断。

### 方法论 13：LLM 语境漂移的识别与防范（新增）

**症状**：LLM 面对超短 / 模糊需求时，会**滑向训练分布里最常见的模式**。

**本周案例**：
| 用户输入 | 预期（紫鸟平台）| AI 实际输出（脱轨）|
|---|---|---|
| "财务的权限" | 紫鸟访问策略模板匹配 + 紫鸟模块追问 | ERP 总账 / 银企直连 / HR 薪酬 / CRM / 代码仓库 |
| "运营做个权限" | 紫鸟店铺运营模板匹配 | 用户数据 / 订单/交易数据 / 内容管理 / 业务数据 |

**识别信号**：输出里出现**当前产品域不存在**的词汇。盘点一下本域实际词汇，再看 AI 输出有没有出现域外词——出现就是脱轨。

**防范手段**：
1. 在 Skill/system prompt 里加一张"输出锚定"章节
2. 明确列 ❌ 禁止词清单（跨行业 RBAC / ERP / HR / CRM / 代码 / 设计等）
3. 同时列 ✅ 合法词清单（本域的模块、页面、概念）
4. **两份清单放在一起做对照**，比单独列其中一份效果强

**底层原理**：LLM 默认回到训练分布最浓的区域，正向引导力度有限，必须用负向约束把它拉回本域。

**适用场景**：任何领域窄、需要强边界感的 AI 产品（比如本项目访问策略助手、客服机器人、垂直分类器）。

### 方法论 14：MVP 路径的 YAGNI 判断——信任原数据的语义信号（新增）

**场景**：做 AI 理解层时，要不要给原数据补前置标注？

**决策矩阵**（以本周 62 个模板为例）：

| 维度 | 全量补标注 | 不补（MVP 路径） |
|---|---|---|
| 工时 | 60 × 10 分钟 = 10 小时 | 0 |
| 准确率提升 | 乐观估计 +5-10 个百分点 | 基线 |
| 冗余度 | 高（name/desc 已含大部分信息）| 低 |
| 迭代成本 | 模板增加要重新标注 | 0 |
| **验证结果** | 未测 | **22/22 命中** |

**判断框架**：
1. 问"**原数据里现有字段已经提供了多少语义信号？**"
   - 模板名是人写的 → 包含核心角色/功能词
   - 描述是人写的 → 包含应用场景
   - URL 可反查 RAG → 映射到中文描述和别名
2. 三者加起来够用就不补；不够用再补**最少**的那条（比如只给命中率差的 5 条模板补）
3. 绝不一开始就"全量前置标注"——大概率 YAGNI

**深化已有框架**：这条和第五周"方法论设计前要先看数据分布"是同一家族——
- 第五周：**看数据再设计维度**（避免设计出 79% 无信息量的 WHO 维度）
- 本周：**看数据再补标注**（避免 10 小时冗余标注）

共同点：**你担心的事可能本来就不存在**，先跑一版再看。

### 方法论 15：静默失败 API 的诊断三原则——假设驱动 / 单变量对照 / 独立校验（新增）

**场景**：提交 API 返回 `ret:0, success`，但实际效果不符合预期（字段未入库、配置被静默裁剪、默认值覆盖）。这类"非报错型失败"是后端 API 最贵的 bug，因为它**把"提交成功"和"意图达成"脱钩了**。

**三原则**：

**原则 1 · 假设驱动**
- 每个诊断步骤必须明确提出**可证伪的假设**："如果 X 是真因，则实验 Y 应该让现象变化；如果 Y 没变化，X 就被证伪"
- 反例：只解释现象（"这可能是因为...也可能是因为..."）→ 没有停止条件，无限延伸

**原则 2 · 单变量对照**
- 复制上一轮失败的 payload，只改**一个字段**，其他完全照抄
- 连续测试 3-5 轮后，差异点被机械定位
- 反例：一次改 3 个字段"看看哪个有效"→ 回归不可解释

**原则 3 · 独立校验**
- 提交后立刻用**独立读接口**（detail/list）回查实际持久化的值
- 不信 add 返回的 `ret:0`，只信 detail 查出来的字段
- 反例：只看 add 返回就宣告成功 → 2 小时后才发现 configs 是空的

**本周实战（2026-05-09 访问策略 API 对接）**：
- ① 怀疑 `configs[].id` 错 → 删掉重提 → 仍空（假设 1 证伪）
- ② 怀疑模式冲突 → 扫全量规则看对照 → 156 黑名单共存（假设 2 证伪）
- ③ 怀疑 `preset_type` → 传 `preset_type:1` → 被后端覆盖回 0（假设 3 证伪）
- ④ 换 url_id → 成功（假设 4 成立：url_id 无效是真因）

四轮耗时 2 小时，比"改三个字段一起试然后瞎蒙"效率高一个数量级。

**可迁移场景**：任何后端接口对接、任何"提交成功但行为异常"的诊断、任何第三方系统联调。

### 方法论 16：跨环境 ID 污染的识别与防范（新增）

**定义**：从环境 A（例：生产抓包、历史文档、其他租户）拷贝一个 ID，直接在环境 B（例：dev）里使用，而环境 B 根本没有这个资源。后端通常不做跨环境校验 → 行为看似合法，实际静默失败。

**典型特征**（符合 3/4 以上即高度怀疑）：
- ID 是从**别处**（生产抓包 / 文档样例 / 同事发的聊天记录）复制来的
- ID 格式**无拼写错**（如 17 位时间戳、UUID、自增数字），看起来完全合法
- 后端接受该 ID 不报错，但相关字段**未生效**
- 同环境其他 ID 工作正常 → 排除了 API 整体故障

**识别 checklist**：
1. 这个 ID 是在**当前环境**注册的吗？
2. 能在当前环境的 `/list` 或 `/detail` 接口里查到吗？
3. 如果答不上来 → 先调 list/detail 验证，再使用

**本周实战**：`url_id=17237191297447`（mooncell.com）——从**生产环境** hqyTest 抓包来的，dev 环境没这个资源。所有用它建的规则 configs 都被静默丢弃。诊断走了 4 轮假设才绕回这个根因。

**预防习惯**：
- 文档/memory 里记 ID 时，**必须标注环境来源**（dev/prod/哪个租户）
- 从别处复制 ID 使用时，**先在目标环境 list/detail 验证存在**
- 跨环境 pipeline（如 dev → prod 迁移）的 ID 映射必须做**显式翻译表**

**可迁移场景**：任何多环境系统（dev/staging/prod）、多租户 SaaS、微服务跨服务 ID 传递、数据迁移。

### 方法论 17：从前端抓包逆推后端模块路径的通用规律（新增）

**问题**：没有后端 API 文档、没法直接问开发，想知道某个功能对应的 API 路径和字段——怎么办？

**操作步骤**（三段式）：

**① 前端抓包取 URL 样本**
- F12 → Network → 筛 Fetch/XHR
- 触发目标功能（点按钮、提交表单）
- 记下请求 URL：一般形如 `/api/v5/security/<module>/<action>`

**② 去版本前缀，直连后端**
- 前端路径经过代理层，带有版本前缀（v1 / v5 等）
- dev 直连后端时，去掉版本段：`/api/v5/security/xxx` → `/api/security/xxx`
- 空 body 发一次 POST：
  - 返回 `{"raw":"no module"}` → 路径错，换模块名或前缀
  - 返回 pydantic validation 错误（`field required`）→ **路径对了**，按错误提示补字段

**③ 按模块名穷举动作**
- 常见动作：`list / add / detail / update / delete`
- 挨个空 body 打，看 pydantic 错误清单，反推每个动作的必填参数

**实战时序（2026-05-09）**：
- 前端抓包 URL = `/api/v5/security/access_rule_url/add`
- 去 v5 直连 dev：`/api/security/access_rule_url/add` → pydantic 错：`name/desc/url_pattern` 必填
- 穷举动作：`/list`（要 page/limit）、`/detail`（要 id）、`/to_regex`（要 urls）、`/delete`（要 url_ids）→ 4 个接口整个模块解锁
- 闭环验证：API 注册 URL → 拿 url_id → 建规则 → detail 查 configs 入库 ✓

**可迁移场景**：
- 老系统没有 API 文档，要对接
- 业务急需某个接口，但开发资源被卡
- 反向工程/迁移/自动化脚本 bootstrap

**局限**：加密信封（生产环境常见）会挡住这条路——只能在 dev/明文环境有效。

**深化已有框架**：和方法论 8"假设验证"同一家族——用最小代价（一次空 body 请求）换取最快的反馈信号（pydantic 错 vs no module）。
