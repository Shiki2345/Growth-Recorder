## 1. 沟通风格

- 用户偏好**简洁而高密度**的回复——读完材料后做提炼，不做冗长复述。
- 用户做总结/周报时维度优先级固定为 **个人思考 > 方法论 > 工作任务**，占比约 40/40/20，不接受流水账式结构。
- 用户日常称 AI 为"Claude"；"成长记录官"只在显式调用 `/GR` skill 时启用，不是跨会话身份。
- 用户喜欢的标准动作链是 **读完 → 提炼 → 更新 memory → 简短汇报**，最后一步控制在几行内。
- 用户对"以后都这样"的偏好已沉淀的有：问题复盘贯穿全程而非单建章节，每个维度追问"为什么"挖根因；技术方案讲业务价值，技术细节点到即止。
- 用户偏好的问句结构是"是不是 X"（假设验证式），不是"怎么 X"（开放式）；让 AI 协助起草跨职能沟通时也按此句式改写。
- 用户反复纠正过的模式：① 不要静默改文件，要先提议再动手 ② 周报不要写成流水账。

## 2. 决策模式

- 用户在"冗余 vs 最小集"之间默认**倾向保留冗余**，特别是当冗余条目承载次要维度信息（marketplace、置信度、来源标签）时；合并/去重作为次选项让用户挑。
- 用户做大体量数据改动前要求先产出**中间评审文件**（markdown 表格），过一眼再写最终产物。

## 3. 产品判断标准

- 用户判断 AI 产品好不好分两层：**生成能力**（看起来对）和**理解能力**（盲测反推），只测生成能力不能宣称"AI 学会了"。
- 用户对 LLM 约束设计按四级强度梯度（软提示→原则→固定话术→禁止清单）从弱到强上行，只在上一级失效时才升级。
- 用户判断"功能边界"时关注两件事：**能力边界**（AI 该不该做）和**语境边界**（AI 用不用本域词）。
- 用户做 demo 打包时按**结构化三问**（runtime Read / 用户点击 / 引用链）筛文件，不凭直觉。
- 用户当前阶段强调**整车视角 > 零件视角**——API 只是轮子，整车是访问策略智能助手；零件 ✓ + deadline 接近时主动切到入口/部署/集成/交付物/完整旅程视角。
- 用户判断扩展可行性时常问四类问题（数据怎么来 / 这一步谁做 / 量级 ×10 / 这人离职能接手吗）。

## 6. 雷区

- 自作主张改文件——必须先指出哪条有问题、提议改法、等用户确认再动手。
- 用"待确认问题 / Open Questions"当占位符代替实际验证清单——用户视为偷懒。
- 把 AI 能力陈述夸大到"学会了 X"——用户要求按三层证据严格分级。
- 在大体量改动里直接写最终产物不出中间评审——用户会打断要求先 markdown 评审表。
- 把"功能看起来相似"当成"可以互相替代"的依据——必须按四问机械验证。
- 推 git 仓库前没配 `.gitignore`，把 `.obsidian/` / `.vscode/` / 凭据文件污染进去——用户因此事后清理过一次仓库。
- 用户要求"读不到的文件明确报告，不要猜"。
